# Lec1 - 什么是操作系统？

## 学习目标
学完这份笔记后，你应当能够解释操作系统的定义，描述进程抽象与保护边界，并能够分析为什么现代负载趋势让操作系统既不可替代又更难设计。

## 2. 操作系统到底是什么

### 2.1 核心定义
一个关键定义是：

- **“Special layer of software that provides application software access to hardware resources.”**

这层软件需要同时提供 **易用抽象**、**受保护的共享访问**、**安全/认证能力** 与 **逻辑实体之间的通信机制**。

### 2.2 核心职责
操作系统同时承担三类任务：
- 向应用提供抽象，包括文件系统、进程/线程、虚拟内存/容器、命名系统。
- 管理资源，包括内存、CPU 时间和存储。
- 落地机制与算法，包括调度、并发控制、事务与安全。

### 2.3 硬件/软件接口
操作系统位于应用与硬件细节之间。应用面向 OS 抽象编程，而不是直接面向底层硬件行为。

![硬件/软件接口](./lec01_materials/os_stack_overview.png)

### 2.4 三种角色：Referee、Illusionist、Glue
课程用三个角色理解 OS：
- **Referee（裁判）**：管理保护、隔离和资源共享。
- **Illusionist（魔术师）**：把物理资源抽象成更干净、易用的接口。
- **Glue（胶水）**：提供存储、窗口系统、网络、共享、授权和统一体验等公共服务。

:::remark 问题：“And – what is it not?”
这个问题在追问操作系统的边界。

操作系统不等于某个应用界面，也不等于单一开发工具。只有当一个软件层承担系统级抽象、资源协调和保护隔离职责时，它才属于操作系统层面的核心能力。
:::

## 3. 进程抽象与虚拟化

### 3.1 程序看到的“机器”
一条核心表述是：

- **“Process: Execution environment with restricted rights provided by OS.”**

每个运行中的程序都在自己的进程中执行。因此，应用真正面对的是进程抽象后的执行环境，而不是裸硬件。

![用进程虚拟化机器](./lec01_materials/process_virtualization.png)

### 3.2 进程里有什么
另一条关键定义是：

- **“A process consists of: Address Space; one or more threads of control executing in that address space; additional system state associated with it (open files, open sockets, …).”**

### 3.3 多程序并存时 OS 的视角
从操作系统视角看，系统中会并存多个进程。OS 把硬件接口翻译为应用可用的进程接口，并为每个程序提供独立、受保护的执行上下文。

## 4. 运行、切换、保护与公共服务

### 4.1 进程运行与切换
操作系统通过调度 CPU 时间并进行上下文切换来并发运行多个进程。

### 4.2 保护边界
保护机制之所以是底座，是因为系统必须同时满足两点：
- 进程之间必须隔离。
- OS 必须与用户进程隔离。

即便它们共享同一套物理硬件，这条隔离边界也必须成立。

![OS 保护边界](./lec01_materials/protection_boundary.png)

### 4.3 I/O、交互体验与后台管理
除 CPU/内存复用外，OS 还需要提供通用 I/O 服务、用户可感知的系统体验（窗口与交互风格），以及网络管理、电源管理等后台能力。

## 5. 为什么操作系统既精彩又困难

### 5.1 为什么必须学操作系统
你可能直接构建操作系统，也可能构建复用 OS 思想的系统组件，或者开发依赖 OS 行为的应用。在这三类路径中，OS 理解深度都会直接影响工程决策质量。

### 5.2 Moore 定律与放缓现实
历史上，晶体管密度大约每两年翻倍，性能常被描述为约每 18 个月翻倍。关键变化是增长显著放缓，课程用“从 **2x every 18 months** 走向约 **1.05x every 18 months**”来概括这一趋势。

### 5.3 算力需求增速超过通用算力增速
AI 训练需求增长明显快于传统 CPU 增长。图中给出的趋势线约为 **35x every 18 months**，这会持续拉大需求与供给之间的缺口。

![需求与供给缺口扩大](./lec01_materials/compute_gap.png)

### 5.4 专用硬件有用，但单靠它不够
GPU、TPU 等加速器是必要条件，但不是充分条件。即使有加速器，系统仍会被利用率、调度与编排、通信、内存和软件复杂度等因素共同限制。

### 5.5 内存需求增长极快
模型参数规模增长非常快，图中给出了约 **40x every 18 months** 和 **340x every 18 months** 的两段斜率；相比之下，单芯片内存增长趋势约为 **1.7x every 18 months**。

这种差距会直接造成部署压力，图中明确提示：在大模型尺度下，仅参数存储就可能需要 **1,000+ GPUs**。

![内存需求与单芯片内存的巨大差距](./lec01_materials/memory_gap.png)

### 5.6 其他硬件与系统趋势
还有几条趋势同样会塑造 OS 设计：
- 存储容量持续增长。
- SSD/Flash 预计会逐步占主导。
- 网络容量持续提升。
- 人均计算设备数量已进入“多人均多 CPU”阶段。
- 系统运行跨越非常大的时间尺度范围。

课件中的代表性存储数据点：
- 最大 3.5 英寸 SSD：**100 TB @ $40K ($400/TB)**。
- 最大 3.5 英寸 HDD：**18 TB @ $600 ($33/TB)**。
- 示例 2.5 英寸三星 SSD：**4 TB @ $380K ($95/TB)**。

### 5.7 软硬件栈复杂性持续上升
现代平台同时包含多核、异构加速器、SoC 集成与不断膨胀的软件规模。穷举测试所有设备/环境组合在工程上不可行。现实问题不是“是否有 bug”，而是“bug 有多严重、系统如何约束其影响”。

### 5.8 分布式成为默认执行环境
LLM 训练与服务、大数据分析、科学计算等负载天然走向分布式。互联网规模服务把系统、协议与管理域串成一个统一执行路径。

### 5.9 例子：一次搜索请求里有什么
一次请求可能同时经过 DNS、数据中心入口、负载均衡、索引系统、页面存储、广告系统与模型排序组件。

![示例：一次搜索请求包含哪些系统组件](./lec01_materials/search_query_path.png)

:::remark 问题组：“How do we tame complexity?”
这个问题先描述现实前提：硬件在 CPU ISA、加速器、内存容量、设备类型和网络环境上都高度异构。

然后提出四个核心问题：
1. **“Does the programmer need to write a single program that performs many independent activities?”**
2. **“Does every program have to be altered for every piece of hardware?”**
3. **“Does a faulty program crash everything?”**
4. **“Does every program have access to all hardware?”**

精炼解答：
- 是的，单个程序常常需要并行处理多项独立活动，因此需要线程/事件驱动模型和 OS 调度支持。
- 不应该每种硬件都重写程序，这正是 OS 抽象与稳定接口存在的意义。
- 有故障的程序不应拖垮全局，隔离与故障抑制是必需机制。
- 程序不应直接访问全部硬件，访问必须通过 OS 受控中介来保证安全、公平与正确性。
:::

## 6. Lab 0、边界讨论与主结论

### 6.1 Lab 0
Lab 0 的任务是启动 Pintos、完成调试、熟悉 Kernel Monitor。给定截止时间是 **2025 年 2 月 27 日**。

### 6.2 讨论中的边界问题
你应当能区分：
- Operating systems vs. computer systems vs. system software。
- Operating systems vs. distributed systems vs. networked systems vs. storage systems vs. database systems。

:::remark 讨论问题与简要回答
**问题 1：** 操作系统、计算机系统、系统软件之间的边界是什么？

操作系统是更大“计算机系统”中的核心层；“系统软件”是更宽泛类别；OS 是其中直接负责硬件虚拟化、受保护资源共享与基础执行抽象的部分。

**问题 2：** 操作系统与分布式/网络/存储/数据库系统的边界是什么？

这些方向在工程中会交叉，但 OS 更关注单机抽象、保护与资源控制；分布式/网络/存储/数据库系统更关注跨机器协同、数据放置、一致性与服务级行为。
:::

## 7. 结论
操作系统通过高层抽象驾驭异构硬件，通过保护与协调管理共享资源，并把语言、数据结构、算法和体系结构等多方面知识整合到统一系统中。它既是工程实践的基础层，也是驯服现代复杂性的核心工具。

## 附录 A：Exam Review

### A.1 必会定义
- **Operating system**：位于应用与硬件之间，负责硬件抽象、资源管理和保护机制的软件层。
- **Process**：由 OS 提供、权限受限的执行环境，包含地址空间、线程和 OS 管理状态。
- **Thread**：进程内部可调度的控制流。
- **Isolation**：阻止跨进程及用户态与内核态之间未授权干扰的机制。
- **Context switch**：保存一个执行上下文并恢复另一个上下文，从而共享 CPU 时间。

### A.2 必记事实
- 成绩结构为 5% 参与、30% labs、25% 期中、40% 期末。
- OS 的角色模型是 **Referee + Illusionist + Glue**。
- AI 算力需求增速可能显著高于通用算力增速。
- 专用硬件本身不能消除系统瓶颈。
- 模型内存需求可能远超单芯片内存扩展速度。

### A.3 高价值简答模板
- **为什么需要进程抽象？**
  进程抽象为每个程序提供受保护的执行环境，并用稳定接口屏蔽底层硬件差异。
- **为什么保护机制是操作系统核心？**
  保护机制防止进程互相破坏或破坏内核，从而支持安全共享、故障抑制和多租户可靠性。
- **为什么加速器本身不够？**
  端到端性能还取决于内存、通信、调度、利用率和软件架构，而不只取决于加速器吞吐。

### A.4 常见误区
- 把操作系统等同于用户界面。
- 讨论性能时忽略隔离与保护。
- 误以为算力增长会自动解决内存与通信瓶颈。
- 只看局部组件，不追踪完整请求路径。
