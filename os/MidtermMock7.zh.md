# 期中考试模拟试题 7：概念与内容判断题（Lec1-Lec13）

## 试卷说明

- 共 50 道判断题。
- 请判断每个命题“正确”或“错误”。
- 本套题故意提高了混淆度，重点考察概念边界、具体语义、实现细节和论文结论。

## 一、判断题

### 判断题 1

操作系统可以被准确地等同于图形用户界面，因为用户主要通过 GUI 感知系统功能。

:::tip 答案与解析
答案：错误。

GUI 只是可能的系统服务或用户界面。OS 的核心是硬件抽象、资源管理、保护隔离、系统服务与受控通信。
:::

### 判断题 2

Referee / Illusionist / Glue 三个角色中，Referee 更强调隔离、保护、认证与资源共享规则。

:::tip 答案与解析
答案：正确。

Illusionist 强调抽象和虚拟化，Glue 强调公共服务与协作接口，Referee 强调受保护共享与裁决。
:::

### 判断题 3

“只有 system 可以访问某些资源”这一性质只需要 OS 软件约定即可实现，不需要硬件模式位或特权检查。

:::tip 答案与解析
答案：错误。

可靠保护必须依赖硬件支持，例如 user/kernel mode、特权指令检查、trap/interrupt 入口和地址翻译权限。
:::

### 判断题 4

线程是可调度的执行上下文；地址空间是内存可见范围与权限状态；进程把二者和其他 OS 状态组合成受限执行环境。

:::tip 答案与解析
答案：正确。

这是 Thread、Address Space、Process 三者的核心关系。
:::

### 判断题 5

对某地址执行读写时，结果一定是普通内存读写；如果地址无效，CPU 会自动忽略该访问。

:::tip 答案与解析
答案：错误。

地址访问可能是普通内存、memory-mapped I/O，也可能触发 fault/trap；无效访问不会被可靠地“自动忽略”。
:::

### 判断题 6

Base-and-Bound 的优点是机制简单、检查路径短；缺点之一是通常要求进程占用连续物理区间，难以表达稀疏地址空间。

:::tip 答案与解析
答案：正确。

B&B 是简单保护机制，但不如分页/虚拟内存灵活。
:::

### 判断题 7

装载期重定位比运行时 Base-and-Bound 翻译更适合支持进程运行时移动和动态扩展。

:::tip 答案与解析
答案：错误。

装载期重定位运行时路径简单，但灵活性较差；运行时翻译通过切换 base/bound 更灵活。
:::

### 判断题 8

Interrupt vector 的作用是根据事件编号找到内核处理入口，而不是由用户程序直接提供跳转地址。

:::tip 答案与解析
答案：正确。

受控入口必须由硬件和内核维护的特权向量表决定，否则用户程序可任意跳入内核。
:::

### 判断题 9

定时器中断属于由当前用户指令主动请求服务的同步事件，因此它和系统调用完全相同。

:::tip 答案与解析
答案：错误。

定时器中断是外部异步事件；syscall 是用户程序主动请求 OS 服务；trap/exception 是当前指令导致的同步异常。
:::

### 判断题 10

在典型 Unix 语义中，多线程进程调用 `fork()` 后，子进程初始只包含调用 `fork()` 的那个线程。

:::tip 答案与解析
答案：正确。

这也是多线程程序中 `fork` 危险的重要原因：其他线程消失，但锁和内存状态被继承。
:::

### 判断题 11

`exec()` 成功后会回到旧程序中继续执行 `exec` 后面的语句。

:::tip 答案与解析
答案：错误。

成功的 `exec` 会替换当前进程映像，通常不会返回旧代码；失败时才返回错误。
:::

### 判断题 12

`pthread_join(t, ...)` 等待的是指定线程 `t`，不是“任意一个先结束的线程”。

:::tip 答案与解析
答案：正确。

即使其他线程先结束，join 指定目标未结束时调用者仍会等待。
:::

### 判断题 13

单核机器上没有真正并行，因此不会发生竞态条件。

:::tip 答案与解析
答案：错误。

单核也会有并发交错。定时器中断、阻塞、yield 等会在读-改-写之间切换线程，仍可能产生竞态。
:::

### 判断题 14

锁主要用于保护临界区互斥；信号量除互斥外，还可表达资源计数和等待约束。

:::tip 答案与解析
答案：正确。

有界缓冲区中 `empty/full` 是资源计数约束，`mutex` 才是互斥约束。
:::

### 判断题 15

只要使用普通 load/store，就足以实现可靠的多核锁，不需要 test-and-set、compare-and-swap 等原子读改写指令。

:::tip 答案与解析
答案：错误。

普通 load/store 无法原子完成“检查并占有”。多核同步需要硬件 RMW 原语或等价机制。
:::

### 判断题 16

pipe 的读端看到 EOF 的条件是所有写端描述符都已关闭，而不是“某个写进程退出”。

:::tip 答案与解析
答案：正确。

只要任意进程仍持有该 pipe 的写端 fd，读端通常不会因为无写者而返回 EOF。
:::

### 判断题 17

`accept()` 返回的是新的 connection socket；原 listening socket 继续用于后续连接。

:::tip 答案与解析
答案：正确。

这是 TCP 服务器循环能持续接收多个连接的关键。
:::

### 判断题 18

socket 可以被视为类文件端点并用 `read/write` 通信，但相比 pipe，它还涉及网络命名、连接建立和协议语义。

:::tip 答案与解析
答案：正确。

socket 继承了文件式 I/O 风格，但不仅仅是本机匿名队列。
:::

### 判断题 19

`fread` 只会从底层 fd 读取调用者请求的字节数，绝不会预取更多数据到用户态缓冲。

:::tip 答案与解析
答案：错误。

C 流可能预取更多数据到用户态缓冲，因此混用 `FILE*` 与低层 fd 会造成 offset 和可见性推理困难。
:::

### 判断题 20

关闭一个由 `dup()` 得来的 fd，只移除一个引用；只要还有别名 fd，底层 open file description 通常不会被销毁。

:::tip 答案与解析
答案：正确。

`close` 关闭的是 fd 表项；open file description 在无引用时才释放。
:::

### 判断题 21

一次 `read(fd, buf, 1000)` 成功时一定返回 1000，否则就必定是错误。

:::tip 答案与解析
答案：错误。

成功读取可以少于请求字节数。返回 0 常表示 EOF；失败通常返回 -1。
:::

### 判断题 22

文件描述符是整数句柄而不是内核指针，这有助于维护用户态与内核态之间的隔离。

:::tip 答案与解析
答案：正确。

用户程序不能直接操作内核对象，只能通过受控 fd 和 syscall 请求服务。
:::

### 判断题 23

多线程进程中 `fork()` 后子进程继承可能不一致的锁状态，因此子进程通常应尽快 `exec()`。

:::tip 答案与解析
答案：正确。

其他线程消失但锁状态可能仍显示“已持有”，子进程若执行复杂库代码可能死锁。
:::

### 判断题 24

如果线程从不 I/O、不等待、不 yield，抢占式 OS 仍可通过定时器中断重新获得控制权。

:::tip 答案与解析
答案：正确。

timer interrupt 是抢占 CPU-bound 用户线程的关键机制。
:::

### 判断题 25

上下文切换代码很容易用穷举测试完全验证，因为 CPU 状态空间很小。

:::tip 答案与解析
答案：错误。

上下文切换涉及寄存器、栈、特权级、异步中断和多种交错，状态空间很大，bug 难以穷举测试。
:::

### 判断题 26

Shinjuku 的启示之一是：微秒级 RPC 中，仅做队列均衡仍可能无法解决长请求阻塞短请求导致的尾延迟问题。

:::tip 答案与解析
答案：正确。

work-conserving 不能自动消除长请求 run-to-completion 的队首阻塞；细粒度抢占很关键。
:::

### 判断题 27

ZygOS 通过保留低开销 dataplane 并加入 work stealing，试图同时获得低开销和接近单队列的 work-conserving 行为。

:::tip 答案与解析
答案：正确。

这是 ZygOS 的核心设计取舍。
:::

### 判断题 28

有界缓冲区中，生产者先 `P(mutex)` 再 `P(empty)` 是安全的，因为阻塞时持有锁可以防止消费者破坏缓冲区。

:::tip 答案与解析
答案：错误。

若缓冲区满，生产者持有 mutex 阻塞在 empty 上，消费者无法进入临界区取走元素，系统可能卡住。
:::

### 判断题 29

在信号量有界缓冲区中，`P` 操作顺序通常比 `V` 操作顺序更直接影响安全性与死锁风险。

:::tip 答案与解析
答案：正确。

错误的 `P` 顺序可能导致持锁等待；`V` 顺序更多影响唤醒时机和性能。
:::

### 判断题 30

Mesa 管程语义下，被 `signal` 唤醒的线程恢复执行时，等待条件一定仍然成立，因此 `if` 比 `while` 更合适。

:::tip 答案与解析
答案：错误。

Mesa 语义中 signal 只让等待者进入 ready queue；真正运行前条件可能已改变，必须用 `while` 重新检查。
:::

### 判断题 31

Hoare 管程语义比 Mesa 语义更直观，因为 signal 后等待者立即获得锁和 CPU，但实现成本通常更高。

:::tip 答案与解析
答案：正确。

Hoare 语义推理干净，但真实系统多采用实现成本较低的 Mesa 风格。
:::

### 判断题 32

条件变量和信号量的一个关键区别是：条件变量的 `signal` 通常不保存历史，而信号量的 `V` 会影响计数。

:::tip 答案与解析
答案：正确。

无人等待时条件变量 signal 通常是 no-op；信号量 V 会增加计数，影响未来 P。
:::

### 判断题 33

写者优先读者/写者管程会在已有等待写者时阻塞新读者，这能改善写者等待，但可能导致读者饥饿。

:::tip 答案与解析
答案：正确。

入口条件 `(AW + WW) > 0` 正是该策略的核心。
:::

### 判断题 34

读者/写者管程中，把所有读者和写者都放在同一个条件变量上，一定比双条件变量更高效。

:::tip 答案与解析
答案：错误。

单条件变量可能唤醒错误类型线程，常需 broadcast，带来惊群；双条件变量可定向唤醒。
:::

### 判断题 35

FCFS 对到达顺序敏感，长作业排在队首会造成短作业队首阻塞。

:::tip 答案与解析
答案：正确。

这正是 FCFS 的主要缺点之一。
:::

### 判断题 36

RR 的时间片越小，平均完成时间和吞吐量必定越好。

:::tip 答案与解析
答案：错误。

时间片过小会增加切换开销和碎片化；RR 不保证平均完成时间优于 FCFS。
:::

### 判断题 37

SRTF 在抢占式策略中可作为平均完成时间最优的理论标尺，但真实系统难以精确实现，因为需要未来运行时间信息。

:::tip 答案与解析
答案：正确。

实际系统常用历史预测和启发式近似，还要叠加公平保护。
:::

### 判断题 38

MLFQ 通过历史行为近似区分交互型和 CPU-bound 任务，但可能被程序通过插入无意义 I/O 或 sleep 来博弈。

:::tip 答案与解析
答案：正确。

课堂中的 Othello/`printf` 类例子说明启发式分类可能被策略性利用。
:::

### 判断题 39

Lottery scheduling 长期 CPU 份额与票数大致成比例，但它无法在系统过载时创造额外 CPU 周期。

:::tip 答案与解析
答案：正确。

比例份额不是扩容；过载仍需要准入控制、削峰或增加资源。
:::

### 判断题 40

RR 能给等待时间某种轮转公平，因此它天然能满足所有硬实时 deadline。

:::tip 答案与解析
答案：错误。

实时系统关注绝对 deadline 和可预测性。RR 公平不等于 deadline 可满足。
:::

### 判断题 41

EDF 的核心规则是始终运行绝对截止期最近的活跃任务。

:::tip 答案与解析
答案：正确。

在课程简化模型下，EDF 可用利用率条件进行可调度性判断。
:::

### 判断题 42

CFS 中 `nice` 越高，通常权重越低，获得的 CPU 份额越少。

:::tip 答案与解析
答案：正确。

课程公式为 `Weight = 1024/(1.25)^nice`，nice 增大则权重下降。
:::

### 判断题 43

CFS 的 target latency 足以解决所有切换开销问题，因此不需要 minimum granularity。

:::tip 答案与解析
答案：错误。

可运行任务很多时，target latency 等分会产生极小时间片，minimum granularity 用来避免过度切换。
:::

### 判断题 44

优先级反转中，高优先级任务可能被低优先级锁持有者间接阻塞，同时中优先级任务继续运行放大延迟。

:::tip 答案与解析
答案：正确。

priority donation/inheritance 的目的就是让锁持有者尽快运行并释放锁。
:::

### 判断题 45

死锁一定意味着某些线程饥饿，但饥饿不一定意味着死锁。

:::tip 答案与解析
答案：正确。

死锁是更强的无法前向进展声明；饥饿可能因未来调度变化而结束。
:::

### 判断题 46

资源分配图中只要有环，不管资源是否有多个实例，都必定死锁。

:::tip 答案与解析
答案：错误。

单实例资源中环通常等价于死锁；多实例资源中有环不一定死锁，还需检测是否有线程可先完成并释放资源。
:::

### 判断题 47

Banker 算法的目标是在批准请求前维持 safe state，而不仅是检查当前是否已经 deadlock。

:::tip 答案与解析
答案：正确。

避免策略比检测当前死锁更强，需要前瞻是否仍存在完整安全序列。
:::

### 判断题 48

DRF 的核心是让每个用户的资源份额总和相等，而不是比较 dominant share。

:::tip 答案与解析
答案：错误。

资源份额总和相等更接近 asset fairness，可能违反 share guarantee。DRF 比较 dominant share。
:::

### 判断题 49

Tiresias 使用 `#GPUs × executed time` 作为二维 attained service，是因为只看执行时间会忽略 GPU 数量这一空间维度。

:::tip 答案与解析
答案：正确。

多 GPU 作业运行较短时间也可能已经消耗大量集群服务量。
:::

### 判断题 50

FairRide 为了同时完美满足 isolation guarantee、strategy-proofness 和 Pareto efficiency，会完全避免对诚实用户产生 blocking。

:::tip 答案与解析
答案：错误。

FairRide 接受 near-optimal Pareto efficiency。blocking 可能影响诚实用户，因为系统无法总是区分作弊和合法访问模式变化。
:::
