### 填空大题 1：运行一个程序时 OS 到底做了什么？

某教学 OS 从磁盘加载一个用户程序 `app` 并运行。程序运行过程中发生一次非法地址访问，随后用户又执行一次合法 `read` 系统调用。请根据 OS 的“底线任务”和四个基础概念填空。

```text
Load executable -> build address space -> create initial thread context
-> enter user mode -> app runs -> fault/syscall/interrupt returns to kernel
```

1. OS 运行程序前，需要把可执行文件中的（1）______ 段和数据段装入内存。
2. 进程地址空间通常包含 code/text、static data、（2）______ 和（3）______ 等区域。
3. 初始线程上下文至少要设置 PC、寄存器和（4）______ 指针。
4. 用户程序正常运行时处于（5）______ mode，不能直接执行特权操作。
5. 非法地址访问属于（6）______ / exception，而 `read(fd, buf, n)` 通常通过（7）______ 进入内核。
6. 定时器到期抢占用户程序属于（8）______。
7. 内核根据事件编号查找处理入口的数据结构是（9）______。
8. 进程的定义可以概括为：带受限权限的（10）______，包含地址空间、一个或多个线程以及 OS 关联状态。
9. 简答：为什么“线程封装并发，地址空间封装保护”？（11）______

:::tip 答案与解析
（1）代码 / text。  
（2）heap。  
（3）stack。  
（4）栈 / stack。  
（5）user。  
（6）trap。  
（7）syscall。  
（8）interrupt。  
（9）interrupt vector / exception vector。  
（10）执行环境。  
（11）线程保存 PC、寄存器、栈等可调度执行状态，使同一进程内可有多个控制流；地址空间定义可访问内存集合和权限边界，阻止进程越权访问 OS 或其他进程，因此承担保护与隔离职责。
:::

### 填空大题 2：pThreads 小组讨论与非确定性输出

课堂讨论过 pThreads 例子：主线程创建多个 worker，再逐个 join。考虑如下伪代码，`counter` 初始为 0：

```c
int counter = 0;

void *worker(void *arg) {
    int local = counter;
    local = local + 1;
    counter = local;
    printf("%s:%d\n", (char *)arg, counter);
    return NULL;
}

pthread_create(&t1, NULL, worker, "A");
pthread_create(&t2, NULL, worker, "B");
pthread_join(t2, NULL);
pthread_join(t1, NULL);
printf("final:%d\n", counter);
```

请填空：

1. 程序总共有（1）______ 个线程，其中包括（2）______ 个主线程和（3）______ 个 worker。
2. `pthread_join(t2, NULL)` 等待的是（4）______ 线程；即使 `t1` 先结束，主线程也（5）______ 从这个 join 返回。
3. `worker` 中 `local = counter; local++; counter = local;` 这段读-改-写不是（6）______ 操作。
4. 若两个 worker 都先读到 `counter == 0`，最终 `counter` 可能为（7）______，这类错误称为（8）______ update。
5. 若两个 worker 串行执行，最终 `counter` 应为（9）______。
6. 保护该临界区可使用 `pthread_mutex_lock` 和（10）______。
7. 简答：为什么“打印顺序变化”不一定代表程序错误，但 `counter` 最终值错误代表同步 bug？（11）______

:::tip 答案与解析
（1）3。  
（2）1。  
（3）2。  
（4）`t2`。  
（5）不会。  
（6）原子。  
（7）1。  
（8）lost。  
（9）2。  
（10）`pthread_mutex_unlock`。  
（11）线程调度本来就是非确定的，因此 A/B 谁先打印可以变化；但若程序语义要求两个 increment 都生效，最终值应为 2。最终值为 1 表示两个线程对共享变量的读-改-写发生竞态，丢失了一次更新。
:::

### 填空大题 3：C 流缓冲与混用 I/O 的推理

文件 `foo.txt` 初始内容为：

```text
abcdefghij
```

考虑如下代码：

```c
FILE *f = fopen("foo.txt", "rb");
int fd = fileno(f);
char x[4] = {0};
char y[4] = {0};

fread(x, 1, 3, f);
read(fd, y, 3);
```

请填空：

1. `fopen` 返回的是高层流对象，类型为（1）______。
2. `fileno(f)` 取得的是底层（2）______。
3. `fread(x, 1, 3, f)` 让应用层看到的 `x` 通常为（3）______。
4. 但 `fread` 可能从底层 fd 一次预取（4）______ 3 字节的数据到用户态缓冲。
5. 因此后续 `read(fd, y, 3)` 可能不从文件逻辑位置 3 开始，而从底层 open file description 的（5）______ 开始。
6. 这说明混用 `FILE*` 与 fd 的风险是两套抽象的（6）______ 和 offset 模型可能偏离。
7. 若想让另一个进程可靠读到 `fwrite` 的结果，应显式调用（7）______ 或 `fclose`。
8. `fdopen(fd, "r")` 的作用是把已有 fd 包装成（8）______。
9. 简答：为什么用户态缓冲是有价值的，即使它增加推理复杂度？（9）______

:::tip 答案与解析
（1）`FILE*`。  
（2）文件描述符 / fd。  
（3）`abc`。  
（4）多于。  
（5）当前位置 / 已被预取推进后的位置。  
（6）缓冲。  
（7）`fflush`。  
（8）`FILE*`。  
（9）用户态缓冲能合并小 I/O，减少系统调用次数，同时提供 `fgets`、格式化 I/O、按行处理等便利功能；代价是必须显式管理刷新和避免混用抽象造成状态不一致。

解析：本题不要求给出 `y` 的唯一字符串，因为不同 C 库缓冲策略可能不同；考点是“混用会让底层 offset 和流内逻辑位置偏离”。
:::

### 填空大题 4：协议设计与 socket 生命周期


```c
bind(s, ...);

while (1) {

```

1. `GET <key>\r\n` 属于协议的（1）______。

6. 若每次 `accept` 后 `fork` 子进程处理连接，隔离性（7）______，但进程创建/切换开销（8）______。
7. 若改为线程池，master 线程是任务队列的（9）______，worker 线程是（10）______。
8. 简答：为什么协议必须同时定义 syntax 和 semantics？（11）______

:::tip 答案与解析
（1）syntax / 语法。  
（2）semantics / 语义。  
（3）listening / 监听。  
（4）connection / connected / 已连接。  
（5）协议。  
（6）串行。  
（7）更强 / 较强。  
（8）更高 / 较高。  
（9）生产者。  
（10）消费者。  
（11）只有格式没有含义，双方不知道收到消息后应做什么；只有含义没有格式，双方无法可靠解析消息。网络通信必须同时约定消息长什么样，以及每类消息触发什么行为和状态变化。
:::

### 填空大题 5：事件驱动 ATM 与线程模型的选择

某 ATM 后端有两种实现：

```text
Version T: 每个请求由一个 worker thread 处理。
Version E: 单线程事件循环，把请求拆成 StartOnRequest / ContinueRequest / FinishRequest 回调。
```

请填空：

1. Version T 的好处是代码可按近似（1）______ 控制流书写，阻塞 I/O 可由调度器切换到其他线程。
2. Version T 的风险是多个线程共享 `balance` 时可能发生（2）______ condition。
3. 若两个线程都先读到 `balance=100`，分别写回 130 和 150，最终余额可能为（3）______ 或（4）______，而正确值应为（5）______。

5. Version E 的难点是一次请求逻辑被拆成状态机，必须跨回调保存（7）______ 状态。
6. 如果 Version E 的某个回调执行阻塞磁盘 I/O，事件循环会被（8）______，其他请求无法推进。
7. 新线程启动时常经过 `ThreadRoot`，用于调用用户函数并在线程函数返回后执行（9）______。
8. 简答：什么时候你会选择线程池而不是事件驱动单线程？（10）______

:::tip 答案与解析
（1）顺序 / 串行。  
（2）race。  
（3）130。  
（4）150。  
（5）180。  
（6）锁 / mutex。  
（7）中间 / 请求。  
（8）阻塞。  
（9）退出清理 / `thread_exit`。  
（10）当处理逻辑包含较多阻塞调用、多核并行很重要、代码希望保持顺序风格，且能通过锁/队列控制共享状态时，线程池通常更自然；事件驱动适合大量 I/O 连接且每个回调很短、非阻塞 API 完整的场景。

（1）低开销。  
（2）单队列 / work-conserving。  
（3）remote。  
（4）抢占。  
（5）`#GPUs * executed time`。  
（6）`100`。  
（7）`80`。  
（8）J2。  
（9）CPU。  
（10）memory。  
（11）`2x/18` 或 `x/9`。  
（12）`3y/18` 或 `y/6`。  
（13）`6`。  
（14）`4`。  
（15）`9`。  
（16）`12`。  
（17）`9`。  
（18）尾延迟、系统开销、负载均衡、抢占粒度、异构资源、作业运行时未知、多资源公平、策略性用户行为和机制设计。

解析：DRF 中 `x/9 = y/6`，得 `2x=3y`，即 `x=1.5y`。资源约束为 `2x+y<=18` 和 `x+3y<=18`，代入得 `4y<=18`、`4.5y<=18`，内存约束更紧，`y=4, x=6`。
:::
