## 一、简答题

### 简答题 1：OS 边界讨论与复杂性控制

课堂开场讨论了 **“What is an Operating System? And what is it not?”**。请回答：

1. OS 与 computer system、system software、distributed system、database system 的边界应如何理解？
2. 为什么现代负载趋势让“只靠更快硬件”不是充分答案？
3. 面对系统复杂性，OS 通过哪些方式约束 bug 的影响？

:::tip 答案与解析
OS 是计算机系统中的核心软件层，直接负责硬件抽象、受保护资源共享、执行环境、基础通信与系统服务。computer system 是更大的整体，包含硬件、OS、运行时、应用和网络环境。system software 是更宽泛类别，编译器、数据库、分布式存储也可能属于系统软件，但不一定承担 OS 的保护边界和硬件虚拟化职责。distributed system、database system、storage system 等会复用 OS 思想，但通常是在 OS 之上构造更专门的跨机器或数据管理抽象。

更快硬件不是充分答案，因为瓶颈常出现在端到端路径中：调度、通信、内存容量、加速器利用率、I/O、故障隔离和软件复杂性都会限制最终性能。AI 训练、数据中心服务和异构硬件让资源不再是单一 CPU 时间，系统必须同时管理多种资源和跨机器协作。

OS 约束 bug 的方式包括：进程隔离、用户态/内核态边界、地址翻译与权限检查、受控 syscall/trap 路径、文件描述符这类不暴露内核指针的句柄、资源限额、调度与回收机制。现实目标不是“完全没有 bug”，而是让局部 bug 不轻易破坏内核、其他进程或全局服务。
:::

### 简答题 2：Base-and-Bound 的优缺点与两种实现方式

课堂讨论问过：**“What are the pros and cons of Base and Bound? What are the pros and cons of the two implementation approaches?”** 请分别比较 Base-and-Bound 机制本身，以及装载期重定位和运行时在线翻译两种实现方式。

:::tip 答案与解析
Base-and-Bound 的优点是概念简单、硬件检查路径短、容易解释保护语义：用户程序只能访问 `[base, base + bound)` 范围，越界会 trap。它也能较直接地支持进程隔离和保护 OS。缺点是每个进程通常需要连续物理区间，内存分配更受约束，碎片管理困难；它表达不了稀疏地址空间、共享页、按页权限等更细粒度需求。

装载期重定位是在程序装入时把地址改成实际物理地址。优点是运行时访存路径简单，不需要每次访存加 base；缺点是依赖 relocating loader，灵活性差，进程移动或动态扩展更困难。

运行时在线翻译由硬件在每次访存时检查 bound 并加 base。优点是更灵活，OS 可以切换进程时换寄存器来改变映射；缺点是每次访存都在关键路径上，需要硬件支持，后续若要支持更复杂地址空间，需要更强的虚拟内存机制。
:::

### 简答题 3：为什么需要多线程，以及线程 vs 进程的取舍

课堂 group discussion 中问过：后台线程执行 `ReadLargeFile("pi.txt")`，UI 线程执行 `RenderUserInterface` 时，程序应表现为什么？请用这个例子解释为什么需要线程，并比较“用线程并发”和“用进程并发”的优缺点。

:::tip 答案与解析
理想表现是：大文件读取可能阻塞或等待 I/O，但 UI 仍能继续响应和渲染。线程的价值正在于把一个程序拆成多个控制流，让计算、I/O 等待和用户交互能够重叠推进。线程共享同一地址空间，通信方便，创建和切换成本通常低于进程；同进程内多线程共享 heap、全局变量和打开文件等状态，每个线程有自己的寄存器上下文和栈。

线程的缺点是隔离弱，一个线程的内存错误可能破坏整个进程；共享状态带来竞态、死锁和难复现 bug。进程并发隔离更强，故障影响更容易被边界约束，权限和资源控制也更清楚；缺点是 IPC 更重，创建/切换成本更高，跨进程共享状态需要 pipe、socket、共享内存或文件等机制。

因此，若任务强共享状态且需要低开销协作，线程常更自然；若任务之间信任边界明显、故障隔离重要，进程更合适。
:::

### 简答题 4：为什么不总是用普通文件做 IPC？

课堂讨论问过：**“为什么基于文件的通信有时会浪费？”** 请比较普通文件、pipe 和 socket 作为 IPC 机制的优缺点，并说明协议中的 syntax 与 semantics 分别是什么。

:::tip 答案与解析
普通文件适合持久化、异步交换和事后读取，但对瞬时通信常显得笨重：数据先落到存储抽象，双方还要约定文件名、轮询可见性、清理临时文件，并处理缓冲与一致性问题。它的优点是简单、持久、容易调试；缺点是延迟和语义都不适合很多在线请求。

pipe 是内核中的单向有界队列，适合同机父子或相关进程之间的流式通信。它支持阻塞、EOF、`SIGPIPE/EPIPE` 等明确规则，开销比文件式通信更贴合瞬时数据流。缺点是通常单向、命名能力弱，跨机器不适用。

socket 是通信端点抽象，可跨机器，支持双向连接和 host/port 命名，适合客户端-服务器模型。代价是协议、连接建立、错误处理和并发服务设计更复杂。

协议的 syntax 是消息格式，例如字段顺序、分隔符、编码方式；semantics 是消息含义和状态变化，例如收到 `GET key` 后应返回什么、连接关闭表示什么、错误码如何解释。
:::

### 简答题 5：Too-Much-Milk 的三个方案为什么是课堂经典？

Too-Much-Milk 中，方案 #1 单便签偶发失败，方案 #1.5 先贴便签破坏活性，方案 #3 不对称双便签可行但不理想。请分别说明它们揭示了哪些同步设计原则。

:::tip 答案与解析
方案 #1 的问题是检查与行动之间不是原子的。两个线程都可能检查到“没牛奶、没便签”，然后都去买。它揭示了并发 bug 常依赖很窄的交错窗口，因此“多数运行正确”不等于正确。

方案 #1.5 先贴便签试图避免重复购买，但两人都先贴便签时可能都认为对方会买，最终没人买。它揭示了安全性和活性不同：安全性是“不发生坏事”，活性是“好事最终发生”。只保证不重复买不等于保证需要时有人买。

方案 #3 通过不对称规则让至少一方能安全推进，说明同步协议可以用精细状态和偏序打破对称性。但它不理想，因为代码复杂、难扩展到多线程、等待时忙等消耗 CPU。课堂用它说明：我们需要更一般、更易证明的同步抽象，如锁、信号量、管程和条件变量。
:::

### 简答题 6：读者/写者讨论题的策略推理

某读者/写者管程使用写者优先策略。课堂讨论了以下改动：

1. 读者退出时无条件 `cond_signal(okToWrite)`。
2. 把这个 `signal` 改成 `broadcast`。
3. 读者和写者共用一个条件变量 `okContinue`。

请分析这三种改动分别可能带来什么正确性或性能影响。

:::tip 答案与解析
无条件 `cond_signal(okToWrite)` 通常仍可保持正确性，因为写者醒来后还会在 `while (AW + AR > 0)` 中重新检查条件；但它会造成无效唤醒。例如还有活跃读者时唤醒写者，写者醒来后发现条件仍不满足，又睡回去，增加上下文切换。

把 signal 改成 broadcast 给写者会放大无效唤醒，产生惊群效应。多个写者同时醒来竞争管程锁，最终通常只有一个能进入，其余又睡回去，吞吐和缓存局部性都会变差。

共用一个条件变量可以简化结构，但 `signal` 可能唤醒错误类型线程。例如应该唤醒写者时唤醒了读者，而读者因 `WW > 0` 又睡回去，真正能推进的写者仍睡着。实践中常需要 broadcast 让所有等待者重新检查谓词，但代价是惊群。双条件变量更复杂一点，却能做更精准的定向唤醒。
:::

### 简答题 7：高层 I/O 缓冲的利弊与可见性推理

课堂 discussion 问过为什么在用户态做缓冲。请说明 `FILE*` 用户态缓冲的收益和代价。若一个线程用 `fwrite` 写文件，另一个进程马上用 `read` 读同一文件，为什么可能读不到刚写入的数据？这个问题和 `fork` 后 fd 别名共享有什么不同？

:::tip 答案与解析
用户态缓冲的收益是减少系统调用次数，把小块读写合并成较大的底层 `read/write`，显著降低 syscall 开销；同时高层流 API 提供 `fgets`、格式化 I/O、按行读取等便利功能。代价是可见性和顺序变得更难推理：数据可能已经进入 `FILE*` 缓冲区，但还没有进入内核文件对象。

因此，一个进程 `fwrite` 后若没有 `fflush` 或 `fclose`，另一个进程马上用低层 `read` 可能看不到新数据。修复方式是显式刷新，或者统一使用同一层 I/O 抽象。

fd 别名共享讨论的是多个 fd 或 fork 后父子进程共享同一个 open file description，因此共享 offset。`FILE*` 缓冲讨论的是用户态运行库在 fd 之上又维护了一层缓冲和位置感知。两者都影响“读写位置和可见性”，但一个在内核 open file description 层，一个在用户态流对象层。
:::

### 简答题 8：从 Shinjuku 到 ZygOS 的尾延迟推理

如果一个微秒级 RPC 服务已经使用多队列 dataplane、轮询和 run-to-completion，为什么仍可能无法满足 99 分位 tail-latency SLO？请比较 Shinjuku 和 ZygOS 在解决这个问题时分别强调了什么。

:::tip 答案与解析
多队列 dataplane、轮询和 run-to-completion 可以降低内核和网络路径开销，但不自动保证尾延迟。多队列可能出现负载不均衡：某个核心队列很长，另一个核心空闲，系统不是全局 work-conserving。run-to-completion 还会让短请求排在长请求之后，形成队首阻塞。对微秒级请求来说，毫秒级或粗粒度抢占已经太慢。

Shinjuku 强调高频抢占和中心化调度视角：只做队列均衡不够，还要用足够细的抢占粒度减少长请求对短请求的阻塞。ZygOS 强调低开销 dataplane 与 work-conserving 的结合：保留 share-nothing 网络路径，同时通过 shuffle layer 和 work stealing 让空闲核心能处理其他核心的就绪连接，必要时用 remote syscall 回到 home core 处理连接状态。

二者共同点是：tail SLO 下不能只看平均吞吐，也不能只看单个请求处理开销；排队结构、抢占粒度、负载均衡和服务时间分布都会决定 99 分位表现。
:::

### 简答题 9：不安全态为什么比“当前无死锁”更危险？

某资源管理器在每次请求时只检查“批准后当前资源分配图没有死锁环”，若没有立即死锁就批准。请解释这个策略为什么弱于 Banker 式安全性检查。请区分 safe state、unsafe state 和 deadlocked state。

:::tip 答案与解析
只检查当前是否死锁是事后或局部判断，它可能批准一个请求，使系统进入 unsafe state。unsafe state 并不等于已经死锁，但从该状态出发，某些后续合法请求会把系统推入不可避免的死锁。Banker 式检查更强，因为它在试探分配后要求存在一个完整安全序列：某个线程能先完成并释放资源，然后下一个线程能完成，直到所有线程完成。

safe state 是存在至少一个所有线程都能完成的顺序；unsafe state 是尚未死锁，但无法保证未来所有合法请求都能避免死锁；deadlocked state 是已经有一组线程形成资源等待闭环并无法推进。

因此，“当前没有死锁”不代表“未来还能安全完成”。避免策略的核心是维持 safe state，而不是只避免眼前的死锁。
:::

### 简答题 10：FairRide 的不可能性讨论

FairRide 讨论了 isolation guarantee、strategy-proofness、Pareto efficiency 三个性质。请解释这三个性质，并说明为什么 FairRide 接受 near-optimal Pareto efficiency，而不是同时追求三者完全成立。概率 blocking 为什么可能影响诚实用户？

:::tip 答案与解析
Isolation guarantee 或 share guarantee 表示用户不应比静态隔离份额更差。Strategy-proofness 表示用户通过虚假访问、虚报需求等策略行为不能获得更高收益。Pareto efficiency 表示无法在不伤害任何人的情况下让某个用户更好。

FairRide 的不可能性结论是：一般情况下缓存共享不能同时完美满足这三者。若追求全局缓存效率，用户可能通过虚假请求改变缓存状态并获益；若要阻止策略行为，就需要对“未付费却搭便车”的访问施加 blocking 或 delay，这会牺牲部分 Pareto efficiency。

概率 blocking 可能影响诚实用户，因为系统无法总是区分策略性作弊和真实访问模式变化。某个用户诚实地访问了别人付费缓存的共享文件，也可能被概率阻断。FairRide 接受这一点，用 near-optimal efficiency 换取 isolation guarantee 和 strategy-proofness。
:::

## 二、大题

### 大题 1：shell、pipe、缓冲与进程边界的综合推理

某 shell 执行：

```sh
cat input.txt | grep OS > out.txt
```

设计者写出如下伪代码，忽略错误处理：

```c
int p[2];
pipe(p);

pid_t a = fork();
if (a == 0) {
    dup2(p[1], STDOUT_FILENO);
    execlp("cat", "cat", "input.txt", NULL);
}

pid_t b = fork();
if (b == 0) {
    dup2(p[0], STDIN_FILENO);
    int fd = open("out.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);
    dup2(fd, STDOUT_FILENO);
    execlp("grep", "grep", "OS", NULL);
}

waitpid(a, NULL, 0);
waitpid(b, NULL, 0);
```

请回答：

1. 这段代码缺少哪些关键 `close`？每个进程应该保留哪些 fd？
2. 在当前代码中，`grep` 可能为什么一直等不到 EOF？
3. `dup2` 后原始 fd 是否还能关闭？关闭后会不会影响标准输入/输出？
4. 如果 `cat` 改成一个使用 `FILE*` 且不 `fflush` 的程序，pipe 输出可见性会如何变化？
5. 如果 shell 是多线程程序，为什么 `fork` 后子进程应尽快 `exec`？这和多线程 `fork` 风险有什么关系？

:::tip 答案与解析
1. 父进程在 fork 完两个子进程后应关闭 `p[0]` 和 `p[1]`，因为父进程既不读也不写 pipe。`cat` 子进程应保留 `STDOUT_FILENO` 指向 pipe 写端，关闭 pipe 读端 `p[0]`，并在 `dup2(p[1], STDOUT_FILENO)` 后关闭原始 `p[1]`。`grep` 子进程应保留 `STDIN_FILENO` 指向 pipe 读端、`STDOUT_FILENO` 指向 `out.txt`，关闭 pipe 写端 `p[1]`，并在 `dup2(p[0], STDIN_FILENO)` 后关闭原始 `p[0]`；`dup2(fd, STDOUT_FILENO)` 后也可关闭原始 `fd`。

2. 当前代码中父进程没有关闭 `p[1]`，`grep` 子进程也可能继承并保留了 `p[1]`。只要还有任何写端描述符打开，pipe 读端就不会看到 EOF。即使 `cat` 已经退出，`grep` 仍可能在读空 pipe 时阻塞。

3. `dup2(oldfd, newfd)` 成功后，`newfd` 成为指向同一底层 open file description 或 pipe 端点的新引用。此时关闭原始 `oldfd` 只移除一个 fd 表项，不会影响 `newfd`。因此关闭原始 `p[1]` 不会影响已经重定向好的 `STDOUT_FILENO`，反而能避免 fd 泄漏和 EOF 问题。

4. 若程序向 pipe 对应的 `FILE*` 写数据但不刷新，数据可能停留在用户态缓冲区，暂时没有进入 pipe。由于 pipe 不是普通磁盘文件，最终进程正常退出或 `fclose` 通常会刷新，但若程序长时间运行且不 `fflush`，下游 `grep` 可能暂时读不到数据。若程序异常退出，也可能丢失未刷新的用户态缓冲内容。

5. 多线程进程 `fork` 后，子进程通常只保留调用 `fork` 的那个线程，其他线程消失，但它们可能曾持有运行库锁、malloc 锁、`FILE*` 锁或应用锁。子进程继承了这些锁的内存状态，却没有对应线程来释放它们，容易死锁。因此多线程程序中，子进程 fork 后应尽快只调用 async-signal-safe 操作并 `exec`，避免在不一致状态中运行复杂逻辑。
:::

### 大题 2：读者/写者管程、优先级反转与死锁的联合情景

一个在线文档服务有三类线程：

- Reader：读取文档内容。
- Writer：修改文档内容。
- Indexer：后台索引线程，低优先级，偶尔持有 `indexLock` 后再读取文档。

系统采用写者优先的读者/写者管程，并有如下状态变量：

```text
AR = active readers
WR = waiting readers
AW = active writers
WW = waiting writers
```

某时刻发生如下事件：

```text
t0: R1, R2 已经在读，AR=2
t1: W1 到达
t2: R3 到达
t3: R1 退出
t4: Indexer 持有 indexLock 后尝试读文档
t5: 高优先级 Search 线程需要 indexLock
t6: 中优先级 CPU 线程持续可运行
t7: R2 退出
```

请回答：

1. 从 `t1` 到 `t7`，`AR/WR/AW/WW` 如何变化？谁会被阻塞？
2. 为什么 `R3` 在 `t2` 不能直接进入，即使当前没有活跃写者？
3. `Indexer` 在 `t4` 的行为可能引入什么锁顺序风险？请构造一个会死锁的双锁顺序。
4. `Search` 在 `t5` 遇到什么调度问题？不启用 donation 时严格优先级会选谁？
5. 你会如何改造这个系统，使它同时降低读者饥饿、死锁和优先级反转风险？

:::tip 答案与解析
1. 初始 `AR=2, WR=0, AW=0, WW=0`。`t1` 时 W1 到达，因 `AR>0` 不能写，进入等待，`WW=1`。`t2` 时 R3 到达，写者优先规则通常在 `(AW + WW) > 0` 时阻塞新读者，所以 `WR=1`，R3 等待。`t3` R1 退出，`AR=1`，仍有 R2 活跃，不能唤醒 W1。`t4` Indexer 尝试读文档，也会按读者入口规则被 `WW>0` 阻塞，`WR=2`，同时它已经持有 `indexLock`。`t7` R2 退出，`AR=0`，若 `WW>0`，唤醒一个写者。W1 被唤醒后会使 `WW--, AW++`，开始写入；R3 和 Indexer 继续等待。

2. R3 不能进入是写者优先策略的直接结果。虽然 `AW=0`，但 `WW=1` 表示已有写者等待。如果继续放行新读者，源源不断的读者可能让写者永远等不到 `AR=0`。因此策略有意牺牲读者等待，换取写者进展。

3. Indexer 在持有 `indexLock` 后再尝试进入文档读者/写者管程，形成锁顺序 `indexLock -> docLock`。若另一个线程先持有 `docLock`，再尝试获取 `indexLock` 更新索引，就会出现 `T1` 持有 `indexLock` 等 `docLock`，`T2` 持有 `docLock` 等 `indexLock` 的 circular wait。典型修复是规定全局锁顺序，例如所有线程必须先拿 `docLock` 再拿 `indexLock`，或把索引更新改成异步消息，避免嵌套锁。

4. Search 是高优先级线程，需要 `indexLock`，但该锁由低优先级 Indexer 持有。中优先级 CPU 线程持续可运行时，严格优先级会选择中优先级线程运行，而不是低优先级 Indexer，因为 Search 已阻塞不可运行。这就是优先级反转：高优先级任务间接被中优先级任务延迟。

5. 改造方案可以组合使用：读者/写者策略上加入公平模式或批量交接，例如写者优先但限制连续写者数量，避免读者长期饥饿；锁顺序上建立全局顺序或禁止持有 `indexLock` 时进入文档管程；调度上对互斥锁启用 priority inheritance/donation，使持有 `indexLock` 的 Indexer 临时继承 Search 的高优先级；等待实现上避免高优先级线程自旋，使用阻塞锁；监控上记录长等待和锁依赖图，发现潜在死锁。
:::

### 大题 3：混合调度器设计与负载推理

某单机系统同时运行以下任务：

| 任务 | 类型 | 到达 | CPU 行为 |
| --- | --- | ---: | --- |
| A | 批处理 | 0 | 需要连续 30ms CPU |
| B | 交互 | 0 | 每轮 2ms CPU 后等待 8ms I/O，共 3 轮 |
| C | 实时 | 0 | 周期 10ms，每周期需要 2ms CPU，硬截止期等于周期 |
| D | 低优先级后台 | 0 | CPU-bound，nice=10 |

系统初版使用 RR，`q=10ms`。后来改成“EDF 实时类 + CFS 普通类 + MLFQ 交互启发式”的混合调度器。

请回答：

1. 为什么单纯 RR 不能保证 C 的硬实时 deadline？即使平均 CPU 利用率不高也可能出问题吗？
2. B 为什么适合被 MLFQ 识别为交互任务？这种识别可能怎样被 A 或 D 博弈？
3. CFS 中 D 的 `nice=10` 会如何影响它和 A 的 CPU 份额？为什么 CFS 仍需要 minimum granularity？
4. 若 C 的实时负载增加到周期 10ms、每周期 7ms CPU，普通类任务会发生什么？此时是调策略还是扩容？
5. 请设计一个合理的调度规则优先级顺序，并说明它牺牲了哪些指标。

:::tip 答案与解析
1. RR 提供的是等待时间层面的轮转公平，不理解绝对 deadline。`q=10ms` 时，一个实时任务可能在一个长时间片后才得到 CPU；若它释放后排在 A 或 D 后面，即使总利用率不高，也可能错过 10ms 内的截止期。硬实时关心“每次释放是否按时完成”，不是长期平均 CPU 份额。

2. B 每轮只运行短 CPU burst，随后等待 I/O，符合交互型任务特征。MLFQ 会倾向让这类任务保持高优先级，以改善响应性。博弈风险是：CPU-bound 的 A 或 D 可以故意插入短 sleep、无意义 I/O 或频繁让出 CPU，让自己看起来像交互任务，从而留在高优先级队列。这会污染启发式信号，导致真正交互任务响应变差。

3. CFS 中 `nice=10` 会显著降低 D 的权重，使它相对 nice=0 的 A 获得更小 CPU 份额。按公式 `Weight=1024/(1.25)^nice`，nice=10 权重大约是 1024/9.31，约为 nice=0 的 1/9.31。CFS 仍需要 minimum granularity，因为当普通类任务很多时，按 target latency 等分会产生极小时间片，过度上下文切换会吞掉吞吐。

4. 若 C 每 10ms 需要 7ms CPU，实时类单独利用率已达 70%。普通类只剩约 30% CPU，而且还会受到实时任务释放时的抢占影响。若再有其他实时任务或系统开销，普通类会明显变慢甚至饥饿。此时先做准入控制和策略隔离，但如果业务目标要求同时维持实时和普通任务吞吐，接近饱和时扩容比继续调参更可靠。

5. 一个合理顺序是：先调度已通过准入控制的硬实时 EDF 类；剩余 CPU 给普通类，普通类内部用 CFS 保证比例份额；交互启发式可影响 CFS/MLFQ 的短期响应，但必须受长期公平约束；后台 nice 高的任务获得较低权重。牺牲是：硬实时优先会降低普通任务吞吐；交互响应优化可能牺牲平均完成时间；公平约束会让某些短作业不能像 SRTF 那样极端优先；准入控制可能拒绝新实时任务。
:::

### 大题 4：现代云平台的多资源、公平与策略行为

一个云平台有 12 CPU、24 GB 内存、8 GPU 和一个共享缓存。平台运行三类用户：

- U1：Web RPC 服务，每个实例需要 `<1 CPU, 2 GB>`，对 99 分位延迟敏感。
- U2：训练作业，每个 worker 需要 `<1 GPU, 1 CPU, 4 GB>`，训练时长未知。
- U3：批处理分析，每个任务需要 `<3 CPU, 3 GB>`，可延迟但希望公平。

另有共享缓存，文件 A/B/C/D 大小都为 1，总缓存容量为 2。用户 X 访问 `A=8, B=4`，用户 Y 访问 `A=8, C=4`，用户 Z 访问 `D=6`。

请回答：

1. 为什么这个平台不能只用单资源 max-min fairness？dominant resource 的思路如何帮助 U1/U2/U3？
2. 若只在 CPU/内存上对 U1 和 U3 做 DRF，设 U1 运行 `x` 个实例，U3 运行 `z` 个任务，请计算 DRF 分配。
3. U2 的训练作业未知总时长时，为什么 Tiresias 不直接用 SRTF？若有两个作业 `J1: 4 GPU 已运行 15min`，`J2: 1 GPU 已运行 40min`，2DAS 如何排序？
4. 对 U1 的 RPC 服务，为什么 ZygOS 式 work stealing 可能提高 tail-SLO 下吞吐？什么时候 Shinjuku 式细粒度抢占更关键？
5. 在缓存中，如果 Y 通过虚增 C 访问试图让 C 完整缓存，并继续搭便车 A，FairRide 的 blocking 如何改变它的收益？Z 这种没有共享热点的用户对 fairness 讨论有什么特殊性？

:::tip 答案与解析
1. 单资源 max-min 不能表达不同用户的瓶颈资源。U1 可能受 CPU 和内存共同约束，更关心尾延迟；U2 的 dominant resource 很可能是 GPU；U3 是 CPU-heavy。若只看 CPU 份额，可能忽略 GPU 稀缺；若只看内存，可能让 CPU-heavy 作业不公平。DRF 用 dominant share 比较用户在自己瓶颈资源上的占比，让不同需求向量的用户可以放到同一公平尺度上。

2. 只看 CPU/内存，总资源 `<12 CPU, 24 GB>`。U1 每实例 `<1 CPU,2 GB>`，dominant resource 是内存，因为 `2/24 = 1/12` 与 `1/12` 相等，二者并列；dominant share 可写为 `x/12`。U3 每任务 `<3 CPU,3 GB>`，dominant resource 是 CPU，因为 `3/12 = 1/4` 大于 `3/24 = 1/8`，dominant share 为 `3z/12 = z/4`。

DRF 令：

$$
\frac{x}{12} = \frac{z}{4} \Rightarrow x = 3z
$$

资源约束：

$$
x + 3z \le 12,\qquad 2x + 3z \le 24
$$

代入 `x=3z` 得 CPU 约束 `6z <= 12`，所以 `z <= 2`；内存约束 `9z <= 24`，所以 `z <= 2.67`。CPU 更紧，取 `z=2, x=6`。U1 运行 6 个实例，U3 运行 2 个任务。U1 分得 `<6 CPU,12 GB>`，dominant share `1/2`；U3 分得 `<6 CPU,6 GB>`，CPU dominant share `1/2`。

3. SRTF 需要知道或准确预测剩余运行时间，而深度学习训练作业总时长通常未知，且会受模型、数据、收敛和分布式通信影响。Tiresias 用 attained service 替代未来预言，采用 `#GPU * executed time`。`J1` 的 2DAS 为 `4*15=60 GPU-min`，`J2` 为 `1*40=40 GPU-min`，因此 J2 attained service 更小，应排在 J1 前。这个排序比只看运行时间更公平，因为 J1 已消耗更多 GPU 总服务。

4. U1 的 RPC 服务受 99 分位延迟约束。ZygOS 式 work stealing 可以缓解多队列负载不均衡，让空闲核心处理其他核心队列中的 ready connection，提高 tail-SLO 下可承载负载。若请求服务时间差异很大，短请求经常被长请求挡住，仅 work stealing 仍不足；此时 Shinjuku 式更细粒度抢占能减少长请求 run-to-completion 造成的队首阻塞，更关键。

5. 诚实时，X 和 Y 共享 A，各自还希望缓存私有 B/C 的一部分。若 Y 作弊让 C 完整缓存，并仍搭便车访问 A，普通 max-min 可能让 Y 得到 `A` 的完整命中加 `C` 的完整命中，从而收益上升。FairRide 会对 Y 访问自己未付费缓存的 A 做 blocking。若 A 由 X 代表缓存，且 `n_A=1`，Y 对 A 的允许概率为 `1/2`，A 的期望命中贡献减半，作弊收益被抵消。

Z 只访问 D，没有共享热点。它的公平问题更接近 isolation guarantee：至少不应因为 X/Y 在共享 A 上获得效率收益而被挤到低于静态份额。Z 也说明缓存公平不是简单“谁访问多谁拿多”，还要考虑没有共享机会的用户不能被共享用户集体压制。
:::
