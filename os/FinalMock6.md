- 题型：简答题 20 题。
- 覆盖范围：主要针对 Lec1-Lec13，即操作系统基本抽象、进程/线程、文件与 IPC、同步、调度、死锁和现代调度案例。

## 一、简答题

### 简答题 1：为什么系统调用入口不能由用户程序随便指定？

用户程序执行系统调用时，为什么硬件/OS 要通过预先设置好的 trap/interrupt vector 进入内核，而不能让用户程序直接跳到任意内核地址？

:::tip 答案与解析
- 系统调用会切换到内核态，若入口地址由用户任意指定，用户就可能绕过权限检查。
- trap/interrupt vector 是受保护的控制转移表，只能由内核设置。
- 固定入口让内核能先保存现场、验证 syscall number 和参数，再决定是否执行特权操作。
- 这体现了 dual mode 的核心：用户可以请求服务，但不能控制内核从哪里开始执行。
:::

### 简答题 2：Base-and-Bound 如何发现越界访问？

在 base-and-bound 机制中，用户程序访问虚拟地址 `vaddr` 时，硬件如何判断该访问是否合法？

:::tip 答案与解析
- 硬件检查 `vaddr < bound` 是否成立。
- 若合法，物理地址为 `base + vaddr`。
- 若不合法，硬件触发异常进入内核，通常终止进程或发送 fault。
- `base` 和 `bound` 必须只能由内核修改，否则用户可以伪造自己的可访问范围。
:::

### 简答题 3：线程共享什么，不共享什么？

同一个进程中的两个线程，哪些状态通常共享，哪些状态通常私有？

:::tip 答案与解析
- 共享：地址空间、heap、全局变量、代码段、打开文件表等进程级资源。
- 私有：寄存器、程序计数器、栈、线程控制块中的调度状态。
- 因为共享地址空间，所以线程通信便宜，但也容易产生 data race。
- 因为有私有栈和寄存器，所以每个线程能停在不同函数调用位置。
:::

### 简答题 4：`fork()` 的返回值为什么能区分父子进程？

`fork()` 成功后，父进程和子进程都从 `fork()` 返回。它们为什么能执行不同分支？

:::tip 答案与解析
- `fork()` 会复制当前进程状态，让父子从同一个程序位置继续。
- 内核刻意设置两个返回值：父进程中返回子进程 pid，子进程中返回 0。
- 程序通常用 `if (pid == 0)` 判断自己是否是子进程。
- 这是同一份代码分裂成两个控制流的关键接口设计。
:::

### 简答题 5：`exec()` 成功后为什么通常不会返回？

调用 `exec()` 成功后，为什么后面的旧程序代码通常不会继续执行？

:::tip 答案与解析
- `exec()` 会用新程序替换当前进程的地址空间。
- 旧代码、旧 heap、旧栈上的大部分用户态内容都不再是原来的程序映像。
- 成功后 CPU 从新程序入口开始执行，而不是回到旧调用点。
- 只有 `exec()` 失败时，才会返回到旧程序并设置错误信息。
:::

### 简答题 6：`fork()` 后文件 offset 为什么可能共享？

进程先 `open("a.txt")` 得到 `fd`，再 `fork()`。如果子进程先 `read(fd, 10)`，父进程随后 `read(fd, 10)`，父进程为什么通常不会从文件开头读？

:::tip 答案与解析
- `fork()` 会复制 fd table，但 fd table entry 指向同一个 open file description。
- open file description 中保存当前 offset、打开模式等状态。
- 子进程 `read` 会推进共享 offset。
- 父进程之后从更新后的 offset 继续读，而不是从 0 开始。
:::

### 简答题 7：Pipe 的 EOF 什么时候出现？

一个 pipe 有多个进程继承了写端。读端 `read()` 什么时候会返回 EOF？为什么只是某个写进程退出还不够？

:::tip 答案与解析
- 只有所有写端引用都关闭后，读端读空 pipe 才会看到 EOF。
- 如果还有任何进程持有写端 fd，内核必须认为未来仍可能有人写入。
- `fork()` 后父子都会继承 pipe fd，所以不需要的一端必须及时 `close`。
- 忘记关闭多余写端，是管道程序卡住的常见原因。
:::

### 简答题 8：上下文切换时为什么必须保存寄存器？

线程 A 被切换出去，线程 B 开始运行。为什么只保存 A 的程序计数器还不够？

:::tip 答案与解析
- 程序计数器只说明下一条指令在哪里。
- 通用寄存器保存局部变量、中间计算结果、函数参数等活跃状态。
- 栈指针决定当前调用栈位置，必须保存和恢复。
- 若不保存这些寄存器，A 之后恢复运行时会看到被 B 覆盖的执行状态。
:::

### 简答题 9：调度器如何重新获得 CPU？

一个用户线程正在运行时，调度器为什么不是“自然地一直在运行”？它通常通过哪两类事件重新获得控制权？

:::tip 答案与解析
- 调度器选中用户线程后，CPU 实际在运行用户代码。
- 内部事件：线程主动进入内核，例如系统调用、阻塞 I/O、退出、yield。
- 外部事件：硬件中断，特别是 timer interrupt，用于抢占长时间运行的线程。
- 没有这些控制转移，单个用户线程可能长期占住 CPU。
:::

### 简答题 10：`x++` 为什么不是原子操作？

两个线程同时执行 `x++`，为什么最终结果可能只增加 1？

:::tip 答案与解析
- `x++` 通常包含读内存、加一、写回内存多个步骤。
- 两个线程可能都先读到旧值 `x = 5`。
- 它们各自计算 `6`，再分别写回 `6`。
- 两次递增的效果丢失一次，这就是 lost update 类型的 race condition。
:::

### 简答题 11：有界缓冲区中为什么先 `P(empty)` 再拿 mutex？

生产者-消费者问题中，生产者通常先等待 `empty`，再进入 mutex 临界区。为什么反过来可能导致问题？

:::tip 答案与解析
- `empty` 表示是否还有空槽；mutex 保护缓冲区元数据。
- 若生产者先拿 mutex，再发现没有空槽而阻塞在 `empty`，它会一直持有 mutex。
- 消费者需要 mutex 才能取走元素并释放 `empty`。
- 于是消费者进不来，生产者也等不到空槽，可能形成死锁。
:::

### 简答题 12：Test-and-Set 自旋锁在单 CPU 上为什么危险？

在单 CPU 系统中，如果线程 H 自旋等待一个被线程 L 持有的锁，为什么可能比阻塞等待更糟？

:::tip 答案与解析
- 单 CPU 上同一时刻只能有一个线程运行。
- H 自旋时消耗 CPU，但锁持有者 L 没有机会运行并释放锁。
- 若 H 优先级更高，还可能一直压制 L，造成严重优先级反转。
- 阻塞等待会让出 CPU，使持锁线程有机会推进。
:::

### 简答题 13：Mesa 条件变量为什么要用 `while`？

在 Mesa monitor 语义下，为什么等待条件变量时推荐写 `while (!condition) wait(cv)`，而不是 `if (!condition) wait(cv)`？

:::tip 答案与解析
- Mesa 语义中，`signal` 只是把等待线程放回 ready queue，不会立即把锁交给它。
- 被唤醒线程真正重新获得锁之前，条件可能已经被其他线程改变。
- 也可能存在 spurious wakeup 或多个等待者竞争同一资源。
- 因此醒来后必须重新检查条件，用 `while` 保证正确性。
:::

### 简答题 14：写者优先读写锁为什么要阻止新读者插队？

在写者优先的 reader-writer lock 中，一旦有 writer 等待，为什么后来的 reader 通常不能继续进入？

:::tip 答案与解析
- 如果新 reader 可以不断进入，已有 writer 可能一直等不到所有 reader 清空。
- 写者优先会记录 `waiting_writers`，让后来的 reader 在 writer 等待时阻塞。
- 已经在读的 reader 可以完成，但新 reader 不再插队。
- 这样能避免 writer starvation，但会牺牲一部分读并发。
:::

### 简答题 15：FCFS 的 convoy effect 是什么？

为什么一个长 CPU-bound 任务排在前面，会让 FCFS 的平均等待时间变差？

:::tip 答案与解析
- FCFS 不会抢占当前任务。
- 长任务先运行时，后面许多短任务必须一起等待它完成。
- 短任务本来可以很快完成，却被长任务“车队”拖住。
- 这会显著增加平均等待时间和交互响应时间。
:::

### 简答题 16：时间片过小的 Round Robin 有什么代价？

RR 调度中，时间片越小响应越快。为什么不能无限减小时间片？

:::tip 答案与解析
- 每次时间片结束都可能触发上下文切换。
- 上下文切换本身不做用户任务的有效工作。
- 时间片太小会让 CPU 花大量时间在调度和切换上。
- 因此 RR 要在响应性和切换开销之间折中。
:::

### 简答题 17：MLFQ 为什么需要 periodic boost？

多级反馈队列中，为什么要周期性把任务提升回高优先级队列？

:::tip 答案与解析
- 长 CPU-bound 任务会逐渐被降到低优先级。
- 如果高优先级队列持续有交互任务，低优先级任务可能长期得不到运行。
- periodic boost 把所有任务重新提升，给低优先级任务恢复机会。
- 它主要用于缓解 starvation，也能让策略重新适应任务行为变化。
:::

### 简答题 18：全局资源顺序为什么能预防锁死锁？

若系统规定所有线程必须按同一个全局顺序申请锁，例如 `A < B < C < D`，为什么这能破坏死锁的 circular wait 条件？

:::tip 答案与解析
- 假设存在 circular wait：每个线程持有某个锁，又等待环中下一个锁。
- 由于所有线程都按全局顺序申请锁，等待的锁必须比自己已持有的锁“更大”。
- 沿着等待环走一圈，会得到锁顺序严格递增。
- 但环回到起点又要求起点锁大于自己，矛盾。
- 因此 circular wait 不可能出现，锁死锁被预防。
:::

### 简答题 19：CFS 为什么选择最小 `vruntime` 的任务？

Linux CFS 中，每个 runnable task 都维护 virtual runtime。为什么调度器倾向于选择 `vruntime` 最小的任务运行？

:::tip 答案与解析
- `vruntime` 近似表示任务已经获得的加权 CPU 服务量。
- `vruntime` 越小，说明它相对自己的权重拿到的 CPU 越少。
- 选择最小 `vruntime` 可以让落后的任务追上来。
- 权重较高的任务 `vruntime` 增长更慢，因此长期获得更大 CPU 份额。
:::

### 简答题 20：DRF 中的 dominant share 是什么？

在多资源公平分配中，为什么 DRF 不直接比较某个任务拿到的 CPU 数量，而要比较 dominant share？

:::tip 答案与解析
- 一个任务可能同时消耗 CPU、内存、GPU 等多种资源。
- dominant share 是该任务在所有资源中占比最大的那一项。
- DRF 让不同用户的 dominant share 尽量公平增长。
- 这样能处理“有人主要吃 CPU、有人主要吃内存”的异质需求，比只看单一资源更合理。
:::
