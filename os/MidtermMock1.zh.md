## 一、简答题

### 简答题 1：操作系统的边界、角色与现代挑战

请用自己的话定义操作系统，并解释为什么“操作系统不等于用户界面，也不等于任意系统软件”。随后分别说明 **Referee、Illusionist、Glue** 三个角色各自解决什么问题，并结合 AI/数据中心/异构硬件趋势说明为什么现代 OS 设计变得更难。

:::tip 答案与解析
操作系统是位于应用与硬件之间的软件层，负责向应用提供受控硬件访问、抽象、资源管理、保护隔离和通信机制。用户界面只是可能由 OS 或系统软件提供的交互入口，不能代表 OS 的全部；编译器、数据库、分布式存储等也属于更宽泛的系统软件或计算机系统组件，只有当一个软件层承担硬件虚拟化、受保护资源共享和系统级执行抽象时，才进入 OS 的核心语境。

三个角色可以这样理解：
- **Referee**：负责保护、隔离、认证和公平共享，例如阻止一个进程破坏另一个进程或内核。
- **Illusionist**：把复杂物理资源虚拟化成更干净的抽象，例如进程、文件、虚拟 CPU、地址空间。
- **Glue**：提供公共服务和统一协作接口，例如文件系统、网络、窗口系统、授权、IPC。

现代 OS 更难，是因为算力需求、模型内存需求、网络规模和软硬件栈复杂度增长很快。GPU/TPU 等加速器很重要，但不能单独解决端到端问题；利用率、调度、通信、内存放置、故障隔离和分布式协作都会成为瓶颈。因此 OS 既要保持抽象简单，又要面对高度异构和跨机器的真实执行环境。
:::

### 简答题 2：四个基础 OS 概念与受控模式切换

请解释 Thread、Address Space、Process、Dual Mode/Protection 四个概念之间的关系。若一个用户程序越界访问地址，随后又正常调用一次系统调用，请说明 Base-and-Bound、trap/syscall、interrupt vector 和用户态/内核态切换各自扮演的角色。

:::tip 答案与解析
Thread 是可调度的执行上下文，包含 PC、寄存器、执行标志和栈等根状态。Address space 是程序可访问地址集合及相关状态，通常包含 code、static data、heap、stack 等区域。Process 是受限权限下的执行环境，封装地址空间、一个或多个线程，以及文件描述符等 OS 状态。Dual mode/protection 则通过硬件模式位、特权指令检查和受控控制转移来保护内核与进程。

在 Base-and-Bound 中，硬件用 `base` 与 `bound` 检查每次地址访问：合法时可概念化为 `physical = base + virtual_offset`；越界时硬件 trap 到内核，内核根据异常类型处理 fault。系统调用则是用户程序主动执行的受控模式切换，请求 OS 服务。syscall、interrupt、exception/trap 都会通过事件编号索引 interrupt/exception vector，找到内核处理入口。进入内核后保存用户态 PC/寄存器等状态，处理完再用受控返回路径恢复用户态执行。
:::

### 简答题 3：线程、进程与 API 语义

比较使用 `pthread_create` 创建线程与使用 `fork`/`exec`/`wait` 管理进程的差异。请说明线程之间共享什么、私有什么，`pthread_join` 的语义是什么，以及为什么 Unix 把“创建进程”和“替换程序映像”拆成 `fork()` 与 `exec()` 两步。

:::tip 答案与解析
同一进程内线程共享地址空间，因此共享 code、static data、heap、打开文件等进程级资源；每个线程有自己的寄存器上下文和栈。`pthread_create` 创建新线程执行 `start_routine(arg)`，函数返回可视为隐式 `pthread_exit`；`pthread_join` 让调用者阻塞，直到目标线程结束并可取得返回值。

`fork()` 复制当前进程，父子初始执行同一程序映像但返回值不同：父进程得到子 PID，子进程得到 0。`exec()` 用新程序替换当前进程映像，成功后通常不会返回旧代码。`wait()` 让父进程等待子进程结束并回收状态。

拆分 `fork` 与 `exec` 的好处是给子进程在 `exec` 前留下准备窗口，例如 shell 可先做 `dup2` 重定向、关闭文件描述符、设置环境变量，再执行目标程序。线程创建更轻量、共享通信方便，但隔离更弱、同步错误风险更高；进程隔离强，但创建、切换和 IPC 成本更高。
:::

### 简答题 4：POSIX 文件抽象、缓冲与 open file description

请解释 `FILE*` 与文件描述符 `fd` 的区别。`open`、`fork`、`dup/dup2`、`read/write`、`fflush` 分别会怎样影响用户态缓冲、fd 表、open file description 和文件 offset？为什么混用 `FILE*` 与 `fd` 容易出错？

:::tip 答案与解析
`fd` 是进程 fd 表中的整数索引，指向内核里的 open file description；open file description 保存文件对象、访问模式、当前 offset 等状态。`FILE*` 是 C 运行库的高层流对象，内部通常包含底层 fd、用户态缓冲区和锁。

`open` 成功后返回一个 fd，并在内核创建或引用 open file description。`read/write` 改变该 open file description 的共享 offset。`fork` 复制父进程 fd 表，父子 fd 会别名到同一个 open file description，因此 offset 共享。`dup/dup2` 在同一进程内创建新的 fd 表项，也指向同一 open file description。`close(fd)` 只删除一个引用，无引用时底层对象才销毁。

`fwrite` 可能只写入用户态流缓冲区，未必立刻进入内核；`fflush` 或 `fclose` 才能强制刷新可见性。混用 `FILE*` 与 `fd` 容易出错，因为两套抽象各自维护缓冲和位置状态，可能出现读写顺序、可见性和 offset 推理不一致。系统代码应严格检查 `open/read/write/fopen/fread` 等返回值。
:::

### 简答题 5：Pipe、Socket 与并发服务器

请说明 POSIX pipe 的两个端点、阻塞规则、EOF 与 `SIGPIPE/EPIPE` 规则。再说明 TCP 服务器中 `socket`、`bind`、`listen`、`accept`、`connect` 的作用，并比较串行服务器、每连接一进程、每连接一线程和线程池模型的取舍。

:::tip 答案与解析
`pipe(int fileds[2])` 创建内核中的单向有界队列，`fileds[1]` 是写端，`fileds[0]` 是读端。缓冲满时写者阻塞，缓冲空时读者阻塞。最后一个写端关闭后，读端读到 EOF；最后一个读端关闭后，写端触发 `SIGPIPE`，若信号被忽略或处理，`write` 通常失败并返回 `EPIPE`。`fork` 后父子继承两个端点，因此必须关闭无用端点，否则 EOF 可能永远不会出现。

TCP 服务器通常先 `socket` 创建端点，`bind` 绑定本地地址/端口，`listen` 进入监听状态，`accept` 从监听 socket 上取出一个新连接并返回连接 socket；客户端用 `connect` 主动连接。TCP 连接可由五元组标识：源 IP、源端口、目的 IP、目的端口、协议。

串行服务器简单但一次只能处理一个连接；每连接一进程隔离性强但创建和切换成本高；每连接一线程共享方便、开销较低，但同步和故障隔离更难；线程池限制并发数量，避免无界建线程导致资源耗尽，是常见工程折中。
:::

### 简答题 6：调度器如何重新获得控制权，以及 Shinjuku 的启示

请解释 PCB/TCB、ready/device queue、dispatch loop、上下文切换和定时器中断之间的关系。一个线程既不 I/O、不等待、也不主动 `yield` 时，OS 如何重新获得控制权？Shinjuku 为什么强调微秒级抢占，而不仅是队列均衡？

:::tip 答案与解析
PCB/TCB 保存进程或线程的调度状态、寄存器上下文、栈指针、资源引用等信息。ready queue 存放可运行实体，device/wait queue 存放等待 I/O 或同步事件的实体。dispatch loop 的核心流程是：选择可运行线程，保存旧线程状态，恢复新线程状态，切换到目标线程执行。

控制权可以通过自愿路径返回内核，例如 I/O、等待锁/条件变量、`yield`；也可以通过外部事件返回内核，例如定时器中断。若线程一直运行且不主动让出，定时器中断会抢占它，使 OS 重新进入调度器。

Shinjuku 的背景是微秒级 RPC 和严格 tail-latency SLO。仅从 `d-FCFS` 改成中心队列或 work stealing 可改善负载不均衡，但不能消除长请求阻塞短请求的队首阻塞。粗粒度抢占如 `PS-1ms` 对微秒级服务时间仍太慢；高频抢占如 `PS-5us` 才能显著改善 99 分位尾延迟。核心启示是调度粒度必须匹配服务时间尺度和尾延迟目标。
:::

### 简答题 7：有界缓冲区与 Too-Much-Milk 的正确性

请写出生产者-消费者有界缓冲区的信号量设计思路，并解释为什么 `P(empty/full)` 与 `P(mutex)` 的顺序重要。再用 Too-Much-Milk 说明“安全性”和“活性”有什么区别，为什么低概率并发 bug 仍然严重。

:::tip 答案与解析
有界缓冲区需要三类约束：
- `emptySlots = bufSize`：生产者进入前等待空槽。
- `fullSlots = 0`：消费者进入前等待可消费项。
- `mutex = 1`：保护队列元数据和入队/出队临界区。

生产者模板是 `P(emptySlots) -> P(mutex) -> Enqueue -> V(mutex) -> V(fullSlots)`；消费者模板是 `P(fullSlots) -> P(mutex) -> Dequeue -> V(mutex) -> V(emptySlots)`。`P` 的顺序是安全性关键：若先拿 `mutex` 再阻塞在 `emptySlots/fullSlots`，等待者会持锁阻止另一方改变条件，可能造成停滞或死锁。`V` 顺序通常不破坏安全性，但会影响唤醒效率。

Too-Much-Milk 中的安全性是“最多一人买奶”，活性是“需要时至少有人买”。单便签方案可能偶发重复购买，破坏安全性；先贴便签或对称双便签可能没人买，破坏活性。低概率并发 bug 仍严重，因为它们依赖特定时序，测试难覆盖，常在生产环境最坏时刻暴露。
:::

### 简答题 8：锁实现、丢失唤醒与 Mesa 管程

比较“关中断实现锁”、纯 `test-and-set` 自旋锁、以及 `test-and-set + guard + 阻塞队列` 三种思路。为什么 sleep/wakeup 中开中断位置错误会导致丢失唤醒？在 Mesa 管程中，为什么 `cond_wait` 外必须用 `while` 而不是 `if`？

:::tip 答案与解析
关中断可以在单处理器内核中保护短小锁元数据更新，但不适合作为通用锁：用户态不能被允许关中断，长临界区会延迟时钟和设备事件，多处理器扩展也差。纯 `test-and-set` 锁简单、可在多核上工作，但等待者会持续自旋，浪费 CPU，并可能放大优先级反转和缓存一致性流量。`test-and-set + guard + 阻塞队列` 只在保护锁元数据的短窗口自旋，锁忙时把线程入队睡眠，释放时唤醒等待者，从而降低忙等成本。

丢失唤醒来自“入队、睡眠、释放/唤醒”之间没有原子化。若入队前开中断，释放者可能看不到等待者而不唤醒；若入队后但真正 sleep 前开中断，释放者可能已把线程标记 ready，而该线程随后又执行 sleep，导致唤醒被覆盖。正确实现要让等待队列操作、睡眠和中断状态恢复由调度路径一致协调。

Mesa 语义下，`signal` 只把等待线程放回 ready queue，发信号线程继续运行。等待线程真正重新拿到锁时，条件可能已被其他线程改变。因此 `cond_wait` 之后必须重新检查谓词，用 `while (condition_not_true) wait(...)`，不能只用 `if`。
:::

### 简答题 9：读者/写者、条件变量与分布式锁

写者优先的读者/写者管程为什么能提升写者进展，但可能造成读者饥饿？请解释 `AR/WR/AW/WW` 的含义，单条件变量与双条件变量的取舍，以及为什么条件变量不能简单等同于信号量。最后说明 Chubby 这类锁服务把锁设计问题扩展到了哪些新目标。

:::tip 答案与解析
`AR` 是活跃读者数，`WR` 是等待读者数，`AW` 是活跃写者数，`WW` 是等待写者数。写者优先策略中，读者入口常在 `(AW + WW) > 0` 时等待，这意味着只要已经有写者等待，新读者即使面对“当前没有活跃写者”也不能继续插队。这样能降低写者等待时间，但若写者持续到达，读者等待时间可能无上界。

双条件变量 `okToRead/okToWrite` 能做定向唤醒，减少唤醒错误类型线程；单条件变量实现简单，但一次 `signal` 可能唤醒仍无法前进的线程，常需要 `broadcast` 让所有等待者重新检查谓词，代价是惊群和更多上下文切换。

条件变量与信号量不同：条件变量不保存历史唤醒，等待线程必须持锁检查谓词；信号量有计数，`V` 可以积累未来 `P` 的通过资格。试图窥探信号量队列来模拟管程通常不安全。

Chubby、ZooKeeper、etcd 这类分布式锁服务把问题从单机互斥扩展为粗粒度协作、高可用、容错、会话失效、租约和一致性语义。分布式锁不只是 `mutex` 放到网络上，还必须面对节点故障、网络分区和恢复后的状态一致性。
:::

### 简答题 10：调度目标、实时性与前向进展

为什么不存在一个在完成时间、吞吐量、公平性和可预测性上都最优的调度策略？请比较 FCFS、RR、SRTF、MLFQ、严格优先级、EDF、CFS 的适用目标与主要风险，并说明优先级反转为什么需要 donation/inheritance。

:::tip 答案与解析
调度目标天然冲突：优化平均完成时间可能偏向短作业并牺牲公平；提高响应性需要更频繁抢占，可能增加切换开销；实时系统关心 deadline，而不是平均值；严格优先级表达重要性，却可能让低优先级任务饥饿。

策略对比如下：
- **FCFS**：简单、开销低，适合吞吐基线，但有队首阻塞。
- **RR**：提供等待时间上界，响应性和等待公平较好，但高度依赖时间片，过小会增加开销。
- **SRTF/SJF**：平均完成时间理论最优或接近最优，但需要预测未来，且短作业持续到达时长作业会饥饿。
- **MLFQ**：用历史行为近似 SRTF，适合交互型任务，但可能被应用通过伪装 I/O 行为博弈。
- **严格优先级**：表达任务重要性强，但有饥饿和优先级反转风险。
- **EDF**：面向实时 deadline，若任务模型满足条件，可用 $\sum C_i/D_i \le 1$ 做可调度性判断。
- **CFS**：通过虚拟运行时间和权重近似比例公平，适合通用 CPU 份额分配，但过载时仍无法消除排队时延。

优先级反转发生在高优先级任务等待低优先级锁持有者，而中优先级任务持续运行压制低优先级任务时。donation/inheritance 会临时提升锁持有者优先级，使其尽快运行并释放锁，恢复高优先级任务的前向进展。
:::

## 二、大题

### 大题 1：Shell 管道、文件重定向与并发服务器

你要实现一个教学 shell，支持命令：

```sh
producer | consumer > out.txt
```

shell 先 `pipe(pipefd)`，再 `fork` 两个子进程。第一个子进程把标准输出接到 pipe 写端并 `exec(producer)`；第二个子进程把标准输入接到 pipe 读端，把标准输出重定向到 `out.txt`，然后 `exec(consumer)`。父进程关闭不需要的描述符并 `wait` 两个子进程。

请回答：

1. `pipe`、`fork`、`dup2`、`exec` 后，fd 表与 open file description 的别名关系如何变化？
2. 父子进程分别应该关闭哪些 pipe 端点？如果父进程忘记关闭写端，会对 EOF 产生什么影响？
3. 为什么 `fork + exec` 的拆分特别适合实现 shell 重定向？
4. 如果 `consumer` 用 `FILE*` 写 `out.txt`，另一个进程立刻用低层 `read` 观察文件，为什么可能看不到刚写的数据？如何修复？
5. 若把 pipe 改成网络 echo 服务，请列出服务端与客户端的核心 socket API，并比较每连接一进程、每连接一线程、线程池三种模型。

:::tip 答案与解析
1. `pipe` 在内核中创建一个单向队列，并在 shell 的 fd 表中返回读端 `pipefd[0]` 与写端 `pipefd[1]`，二者分别引用对应的 open file description。`fork` 后，父子 fd 表被复制，多个进程中的同号或不同号 fd 可能别名到同一 pipe 端点对象。第一个子进程执行 `dup2(pipefd[1], STDOUT_FILENO)` 后，`STDOUT_FILENO` 指向 pipe 写端；第二个子进程执行 `dup2(pipefd[0], STDIN_FILENO)` 后，`STDIN_FILENO` 指向 pipe 读端。第二个子进程再 `open("out.txt", ...)` 得到文件 fd，并用 `dup2(fd, STDOUT_FILENO)` 让标准输出指向该文件的 open file description。`exec` 替换程序映像，但默认保留未设置 close-on-exec 的 fd，因此重定向会延续到新程序。

2. 第一个子进程应关闭 pipe 读端，以及完成 `dup2` 后不再需要的原始写端。第二个子进程应关闭 pipe 写端，以及完成 `dup2` 后不再需要的原始读端。父进程在 fork 完两个子进程后应关闭自己持有的读端和写端。如果父进程忘记关闭写端，即使 producer 已退出，consumer 侧也可能认为仍存在写者，读不到 EOF，而是在 pipe 空时继续阻塞。

3. `fork + exec` 给子进程留下了“执行新程序之前”的准备窗口。shell 可以在子进程里先设置 fd、`dup2` 重定向标准输入输出、关闭无用 fd、设置环境变量，再 `exec` 目标程序。若只有一个合并式 `CreateProcess` 风格接口，也能实现，但接口必须承载更多参数和状态准备逻辑。

4. `FILE*` 有用户态缓冲，`fwrite` 可能只写入流缓冲区，尚未通过底层 `write` 进入内核文件对象。另一个进程用低层 `read` 观察时，可能看不到这部分数据。修复方式是使用 `fflush` 或 `fclose` 建立可见性点，或者统一使用同一 I/O 抽象，避免混用 `FILE*` 和 fd 推理。

5. TCP echo 服务端核心流程是 `socket -> bind -> listen -> accept -> read/write -> close`；客户端核心流程是 `socket -> connect -> read/write -> close`。`accept` 返回新的连接 socket，监听 socket 保持用于后续连接。每连接一进程隔离强、故障影响小，但创建和切换成本高；每连接一线程开销较低、共享方便，但同步错误和故障传播风险更大；线程池限制并发规模，能避免无界建线程带来的资源耗尽，是吞吐、隔离和工程复杂度之间的常见折中。
:::

### 大题 2：ATM 银行服务的并发控制

某银行 ATM 后端有 8 个 worker 线程处理网络请求。请求先进入一个有界队列，worker 从队列取出后更新共享账户。系统支持存款、转账、查询余额和后台审计。账户对象可被多个线程访问，查询很多，写入较少。某高优先级风控线程偶尔需要读取共享状态，但可能等待普通 worker 持有的锁。

请回答：

1. 用一次存款的交错执行说明 lost update 如何发生。
2. 为网络请求队列设计信号量版或管程版有界缓冲区，并解释等待时为什么不能持有队列锁忙等。
3. 账户锁应该用一把全局锁还是细粒度账户锁？若转账需要同时锁两个账户，如何避免死锁？
4. 对余额查询和转账写入，如何用读者/写者管程提升并发？写者优先会带来什么风险？
5. 风控线程遇到优先级反转时，系统应如何使用 priority donation/inheritance？为什么高优先级线程自旋等待可能造成无进展？

:::tip 答案与解析
1. lost update 的典型交错是：初始余额为 `B`。线程 `T1` 读取余额 `B`，准备写回 `B + amount1`；此时 `T2` 读取同一个旧值 `B`，写回 `B + amount2`；之后 `T1` 再写回 `B + amount1`。最终余额只包含其中一次更新，而不是 `B + amount1 + amount2`。问题根源是读、改、写不是原子的。

2. 信号量版请求队列可使用 `emptySlots = N`、`fullSlots = 0`、`mutex = 1`。网络接收线程作为生产者执行 `P(emptySlots) -> P(mutex) -> Enqueue -> V(mutex) -> V(fullSlots)`；worker 作为消费者执行 `P(fullSlots) -> P(mutex) -> Dequeue -> V(mutex) -> V(emptySlots)`。管程版则用一把锁和两个条件变量 `notFull/notEmpty`，满时生产者 `while (full) wait(notFull, lock)`，空时消费者 `while (empty) wait(notEmpty, lock)`。等待时不能持有队列锁忙等，否则真正能改变条件的线程无法进入临界区；即使释放再重拿锁忙等，也会浪费 CPU 并造成调度抖动。

3. 全局账户锁简单、正确性容易证明，但会把无关账户操作串行化，吞吐差。细粒度账户锁并发度更高，但转账涉及两个账户时可能死锁：`T1` 锁 A 等 B，`T2` 锁 B 等 A。常见预防方法是所有转账按全局顺序申请账户锁，例如按账户 ID 从小到大加锁，从结构上打破 circular wait。也可用 `try_lock` 失败释放重试，或封装 `Acquire_both` 并在内部强制统一顺序。

4. 查询是读者，转账/存款是写者。读者/写者管程允许没有写者时多个查询并发，提高查询吞吐；写者需要在没有活跃读者和写者时独占进入。写者优先策略会在 `WW > 0` 时阻塞新读者，避免写者被读者流持续淹没，但若写请求持续到达，读者可能饥饿。实现时要跟踪 `AR/WR/AW/WW`，并谨慎选择 `signal` 与 `broadcast`。

5. 优先级反转流程是：普通低优先级 worker 持有锁，高优先级风控线程等待该锁，中优先级 CPU 任务持续运行，导致锁持有者迟迟不能释放锁。priority donation/inheritance 会把风控线程的高优先级临时传给锁持有者，使其尽快完成临界区并释放锁，然后撤销临时优先级。若高优先级线程自旋等待，它会持续占用 CPU，反而压制低优先级锁持有者运行，系统看似忙碌但有效进展很少，形成活锁或无进展风险。因此等待较长时应阻塞，而不是无限自旋。
:::

### 大题 3：混合负载调度、实时任务与死锁处理

某单 CPU 教学 OS 同时运行批处理任务、交互任务和硬实时任务。先考虑三个同时到达的 CPU burst：`P1=24`、`P2=3`、`P3=3`，忽略上下文切换成本。系统还周期性运行三个实时任务：`T1=(D=4,C=1)`、`T2=(D=5,C=2)`、`T3=(D=7,C=2)`。最后，系统中有五个“用餐律师”线程共享五根筷子，每位律师需要两根筷子才能完成。

请回答：

1. 分别计算 FCFS 顺序 `P1,P2,P3`、SJF 顺序 `P2,P3,P1`、RR（`q=3`，初始队列 `P1,P2,P3`）下三个任务的平均等待时间和平均完成时间。
2. 若目标分别是吞吐量、等待公平、平均完成时间、硬实时 deadline、通用 CPU 份额公平，应分别倾向选择什么策略？这些策略各有什么风险？
3. 判断给定实时任务集在 EDF 下是否可调度，并说明 RR 为什么不能直接保证 deadline。
4. 若 MLFQ 用“短 burst/频繁睡眠”识别交互任务，CPU 密集程序如何博弈它？工程上如何缓解？
5. 五个律师各先拿到一根筷子后都等待第二根，这是死锁还是饥饿？请用死锁四条件和 Banker 思路说明系统应该如何处理。

:::tip 答案与解析
1. FCFS 顺序 `P1,P2,P3`：完成时刻分别为 `24,27,30`，等待时间分别为 `0,24,27`。平均等待时间为 `(0+24+27)/3 = 17`，平均完成时间为 `(24+27+30)/3 = 27`。SJF 顺序 `P2,P3,P1`：完成时刻为 `3,6,30`；按进程看，`P1` 等待 `6`，`P2` 等待 `0`，`P3` 等待 `3`。平均等待时间为 `3`，平均完成时间为 `(30+3+6)/3 = 13`。RR `q=3`：执行序列为 `P1(0-3), P2(3-6), P3(6-9), P1(9-30)`，完成时刻 `P1=30,P2=6,P3=9`；等待时间 `P1=6,P2=3,P3=6`。平均等待时间为 `5`，平均完成时间为 `15`。这说明 RR 改善等待公平，但平均完成时间不一定最优。

2. 吞吐量基线可倾向 FCFS，因为策略开销低，但队首阻塞明显。等待公平可倾向 RR，但时间片过小会拖累吞吐。平均完成时间可倾向 SJF/SRTF 或其近似，但需要预测未来且可能饿死长作业。硬实时 deadline 应倾向 EDF/RMS/DM 等实时策略，并做可调度性或准入判断。通用 CPU 份额公平可倾向 CFS/比例份额策略，但接近饱和时排队时延仍会急剧上升。严格优先级适合表达重要性，但必须防饥饿和优先级反转。

3. EDF 可调度性检查：

$$
\frac{1}{4}+\frac{2}{5}+\frac{2}{7}=0.25+0.4+0.286\approx0.936\le1
$$

因此在该简化模型下可调度。RR 只能给出轮转等待意义上的公平，不关心绝对截止期最近者；任务可能在时间片轮转中错过 deadline，所以公平不等于实时可预测性。

4. MLFQ 依赖历史行为把交互/I/O 型任务留在高优先级，把持续用满时间片的 CPU-bound 任务下调。CPU 密集程序可以插入无意义 I/O 或短暂睡眠，让自己看起来像交互任务，从而留在高优先级。若所有程序都这么做，分类信号失真，高优先级队列被污染。缓解方法包括对长期 CPU 使用做累计约束、周期性 priority boost、防饥饿机制、比例份额约束，以及不要完全信任应用自报。

5. 五个律师各持有一根筷子并等待第二根，是死锁。四条件同时成立：筷子互斥使用；每人 hold and wait；筷子不能被强制抢占；等待关系形成环。它也表现为饥饿，但比普通饥饿更强，因为没有外部干预时无法自行恢复。Banker 避免思路是在每次请求时保持 safe state：若批准某次请求会进入“人人一根、无空闲筷子、无人能先拿到第二根完成”的状态，就应拒绝或延迟。等价地，系统要保证批准后仍存在至少一个律师能最终凑齐两根、完成并释放资源的安全序列。
:::

### 大题 4：现代数据中心中的调度、公平与策略行为

某云平台同时运行四类系统：微秒级 KV/RPC 服务、分布式深度学习训练、CPU/内存多资源作业，以及共享缓存服务。平台目标包括 99 分位 tail-latency SLO、低平均 JCT、多资源公平、以及防止用户通过虚假请求作弊。

请回答：

1. 对微秒级 RPC，为什么单队列/work-conserving 模型常能降低尾延迟？ZygOS 如何同时保留 dataplane 低开销与接近单队列的负载均衡？它与 Shinjuku 的共同问题意识是什么？
2. 对深度学习训练，作业 A 使用 8 GPU 运行 10 分钟，作业 B 使用 2 GPU 运行 30 分钟。按 Tiresias 的 2DAS，谁的 attained service 更小？为什么 2DAS 比单纯 executed time 更合理？模型画像放置解决什么问题？
3. 对多资源分配，总资源为 `<9 CPU, 18 GB>`，用户 1 每个任务需求 `<1 CPU, 4 GB>`，用户 2 每个任务需求 `<3 CPU, 1 GB>`。请用 DRF 计算应运行的任务数，并说明为什么 DRF 比 asset fairness 更稳健。
4. 共享缓存中，文件 A/B/C 大小都为 1，总缓存大小为 2。User 1 访问 `A=10,B=5`，User 2 访问 `A=10,C=5`。诚实时两人各得到 `12.5 hits/s`。若 User 2 虚增 C 使缓存变为 A 属于 User 1、C 属于 User 2，为什么普通 max-min 下 User 2 会获利？FairRide 的概率阻断如何让作弊不再有利？
5. ZygOS、Tiresias、DRF、FairRide 分别体现了现代调度中的哪类核心权衡？

:::tip 答案与解析
1. 单队列模型更接近全局 work-conserving：只要有核心空闲，就更容易从同一工作池取活，减少多队列负载不均衡；在服务时间离散度高或 tail SLO 严格时，这对 99 分位延迟很关键。ZygOS 保留 dataplane 的 share-nothing 低开销网络路径，同时在 shuffle layer 上让空闲 remote core 从 home core 的就绪连接队列 steal 工作；remote core 执行业务逻辑，涉及 home core 网络连接状态时发起 remote syscalls，由 home core 完成 TCP/IP 发送。这让系统在低开销路径上接近单队列的 work conservation。Shinjuku 与 ZygOS 的共同意识是：微秒级服务不能只优化平均吞吐，必须正视队首阻塞、抢占粒度、负载不均衡和 tail-latency SLO。

2. Tiresias 的 2D attained service 为：

$$
\text{2DAS} = \#\text{GPUs}\times \text{executed time}
$$

作业 A 的 attained service 是 `8 * 10 = 80 GPU-min`，作业 B 是 `2 * 30 = 60 GPU-min`。因此 B 的 attained service 更小，按 Least-Attained Service 思想更应优先。2DAS 比单纯 executed time 更合理，因为 GPU 作业的资源占用有空间维度：运行 10 分钟但占 8 张 GPU，已经消耗了比 2 GPU 作业运行 30 分钟更多的集群服务量。模型画像放置解决“是否聚合 GPU”问题：通信敏感模型适合聚合减少同步开销，但过度聚合会造成 GPU 碎片和排队延迟，因此不能一刀切。

3. 用户 1 的 dominant resource 是 memory，dominant share 为 `4x/18`；用户 2 的 dominant resource 是 CPU，dominant share 为 `3y/9`。DRF 令 dominant share 相等：

$$
\frac{4x}{18}=\frac{3y}{9}\Rightarrow 2x=3y\Rightarrow x=1.5y
$$

资源约束是：

$$
x+3y\le9,\qquad 4x+y\le18
$$

代入 `x=1.5y`，CPU 约束给出 `4.5y <= 9`，所以 `y <= 2`；取 `y=2` 得 `x=3`。分配为用户 1 运行 3 个任务，用户 2 运行 2 个任务。此时用户 1 分得 `<3 CPU,12 GB>`，memory dominant share 为 `12/18=66.7%`；用户 2 分得 `<6 CPU,2 GB>`，CPU dominant share 为 `6/9=66.7%`。DRF 比 asset fairness 稳健，因为它比较每个用户的瓶颈资源份额，而不是简单比较资源份额总和；后者可能让某个用户在所有关键资源上都低于应得静态份额，违反 share guarantee。

4. 诚实时，A 被共享缓存，B/C 各缓存一半，所以每人命中贡献为 `10*1 + 5*0.5 = 12.5 hits/s`。若 User 2 通过虚增 C 让 C 获得完整缓存，同时仍搭便车访问 A，普通 max-min 下 User 2 的真实命中贡献为 `A:10 + C:5 = 15 hits/s`，高于 12.5，所以作弊有利。

FairRide 的核心是：未为某文件缓存“付费”的用户访问已缓存文件时，会被概率阻断。阻断概率为：

$$
p(n_j)=\frac{1}{n_j+1}
$$

在作弊状态中，User 2 访问 A 但 A 由 User 1 代表缓存，`n_A=1`，因此阻断概率为 `1/2`，允许概率也为 `1/2`。User 2 对 A 的期望命中贡献变为 `10*1/2=5`，再加上自己缓存 C 的 `5`，总共 `10 hits/s`，低于诚实的 `12.5`。因此作弊不再有利。代价是守规用户在某些共享场景下也可能被部分阻断，FairRide 牺牲完全 Pareto efficiency，保留 isolation guarantee 与 strategy-proofness，并追求 near-optimal efficiency。

5. ZygOS 体现“低系统开销与全局 work-conserving/低尾延迟”的权衡；Tiresias 体现“未知运行时下的 JCT 优化、GPU 空间维度和放置碎片”的权衡；DRF 体现“多资源公平、share guarantee、strategy-proofness 与效率”的权衡；FairRide 体现“共享缓存中的隔离保证、策略无关性与 Pareto 效率不可三得”的机制设计权衡。四者共同说明现代调度不再只是单 CPU 队列选择，而是延迟、异构资源、信息不完整和用户激励的跨层设计。
:::
