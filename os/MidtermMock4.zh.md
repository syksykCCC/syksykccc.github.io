### 选择题 1（单选）：OS 的核心定义

以下哪一项最准确描述了本课程中操作系统的核心定义？

A. 操作系统就是图形用户界面和桌面环境。  
B. 操作系统是位于应用与硬件之间、提供受控硬件访问与资源抽象的软件层。  
C. 操作系统是所有系统软件的总称，包括编译器、数据库和浏览器。  
D. 操作系统只负责启动程序，不负责隔离和资源管理。

:::tip 答案与解析
正确选项：B。

课程定义强调 OS 是应用与硬件之间的特殊软件层，提供抽象、资源管理、保护隔离和通信机制。GUI 只是可能的系统服务之一，不能等同于 OS。
:::

### 选择题 2（不定项）：Dual Mode 的硬件支持

为了可靠支持 user mode / kernel mode，硬件至少需要提供哪些机制？

A. 模式位或等价的特权级状态。  
B. 对特权指令的权限检查。  
C. 受控的 trap / interrupt / syscall 入口。  
D. 只要编译器自动插入检查即可，不需要硬件参与。  
E. 从内核返回用户态时恢复用户 PC 和状态的机制。

:::tip 答案与解析
正确选项：A、B、C、E。

可靠保护必须由硬件强制执行。编译器检查可以辅助，但不能阻止恶意或失控二进制代码绕过规则。
:::

### 选择题 3（单选）：Base-and-Bound 地址翻译

某进程 `base = 0x8000`，`bound = 0x2000`。虚拟地址是从 0 开始的 offset。访问虚拟地址 `0x1fff` 的结果是：

A. 合法，物理地址为 `0x9fff`。  
B. 合法，物理地址为 `0x8000`。  
C. 非法，因为 `0x1fff` 小于 `bound`。  
D. 非法，因为只有 `0x0000` 到 `0x1000` 合法。

:::tip 答案与解析
正确选项：A。

合法范围是 `[0, bound)`，即 `[0, 0x2000)`。`0x1fff` 合法，物理地址为 `0x8000 + 0x1fff = 0x9fff`。
:::

### 选择题 4（单选）：`pthread_join` 的目标

考虑：

```c
pthread_create(&t1, NULL, f1, NULL);
pthread_create(&t2, NULL, f2, NULL);
pthread_join(t1, NULL);
pthread_join(t2, NULL);
```

若 `t2` 比 `t1` 更早结束，主线程会怎样？

A. 先从 `pthread_join(t1)` 返回，因为任意子线程结束都能唤醒它。  
B. 仍然等待 `t1`，因为 join 等待的是指定线程。  
C. 立即跳过两个 join。  
D. 行为未定义。

:::tip 答案与解析
正确选项：B。

`pthread_join(t1, ...)` 等待指定线程 `t1`。即使 `t2` 已经结束，主线程也要等 `t1` 结束后才继续执行。
:::

### 选择题 5（不定项）：`fork` 后的确定偏序

假设使用 `write`，不考虑 stdio 缓冲：

```c
write(1, "X", 1);
pid_t pid = fork();
if (pid == 0) {
    write(1, "C", 1);
} else {
    write(1, "P", 1);
}
write(1, "E", 1);
```

以下哪些说法一定正确？

A. `X` 一定是第一个输出字符。  
B. 子进程中的 `C` 一定早于子进程中的 `E`。  
C. 父进程中的 `P` 一定早于父进程中的 `E`。  
D. `P` 一定早于 `C`。  
E. 总共会输出两个 `E`。

:::tip 答案与解析
正确选项：A、B、C、E。

`X` 在 `fork` 前执行，因此只输出一次且一定最先。父子各自执行一次最后的 `write("E")`，所以有两个 `E`。父子之间的相对顺序不确定，因此 `P` 不一定早于 `C`。
:::

### 选择题 6（不定项）：`exec` 的语义

关于成功执行的 `exec(...)`，以下哪些说法正确？

A. 当前进程的程序映像被新程序替换。  
B. 成功后通常不会返回到旧代码继续执行。  
C. 进程 PID 通常保持不变。  
D. 所有文件描述符一定都会被关闭。  
E. 未设置 close-on-exec 的文件描述符通常会保留。

:::tip 答案与解析
正确选项：A、B、C、E。

`exec` 替换当前进程映像，但进程身份仍是同一个进程。文件描述符默认可跨 `exec` 保留，除非设置了 close-on-exec。
:::

### 选择题 7（不定项）：fd 与 open file description

以下哪些操作会让两个 fd 共享同一个 open file description，从而共享 offset？

A. `int fd2 = dup(fd1);`  
B. `dup2(fd1, 100);`  
C. `fork()` 后父子进程继承同一个已打开 fd。  
D. 对同一路径连续调用两次 `open()`。  
E. `close(fd1)` 后再 `open()` 同一路径。

:::tip 答案与解析
正确选项：A、B、C。

`dup/dup2` 创建 fd 表别名，`fork` 复制 fd 表且仍指向同一 open file description。两次独立 `open` 通常创建两个不同 open file description，offset 不共享。
:::

### 选择题 8（单选）：`FILE*` 缓冲可见性

考虑：

```c
char x = 'c';
FILE *f1 = fopen("file.txt", "w");
fwrite("b", sizeof(char), 1, f1);
FILE *f2 = fopen("file.txt", "r");
fread(&x, sizeof(char), 1, f2);
```

未调用 `fflush(f1)` 或 `fclose(f1)` 时，`x` 的结果最准确说法是：

A. 一定是 `'b'`。  
B. 一定保持 `'c'`。  
C. 可能是 `'b'`，也可能保持 `'c'`。  
D. 一定触发段错误。

:::tip 答案与解析
正确选项：C。

`fwrite` 可能只写入用户态缓冲区，尚未刷新到底层文件；因此另一个流是否读到新数据不应被假设。显式 `fflush` 或 `fclose` 才能建立更明确的可见性。
:::

### 选择题 9（不定项）：pipe 的 EOF 与 `SIGPIPE`

关于 POSIX pipe，以下哪些说法正确？

A. `pipefd[0]` 通常是读端，`pipefd[1]` 通常是写端。  
B. 只要任意一个写端 fd 仍然打开，读端通常不会因为“无写者”而看到 EOF。  
C. 最后一个读端关闭后，继续写可能触发 `SIGPIPE`。  
D. pipe 是双向连接，每个端点都可以同时读写。  
E. 若忽略 `SIGPIPE`，写操作可能以 `EPIPE` 失败。

:::tip 答案与解析
正确选项：A、B、C、E。

Unix pipe 是单向 IPC 队列。EOF 取决于所有写端是否都关闭；所有读端关闭后写会触发 `SIGPIPE`，若信号被处理或忽略，`write` 通常失败并设置 `EPIPE`。
:::

### 选择题 10（单选）：`accept` 的返回值

TCP 服务器已经执行 `socket -> bind -> listen`。调用 `accept(listenfd, ...)` 成功后返回的 fd 是：

A. 原来的监听 socket，之后不能再接受新连接。  
B. 一个新的 connection socket，用于和某个具体客户端通信。  
C. 客户端本地文件的 fd。  
D. 一个 pipe 的写端。

:::tip 答案与解析
正确选项：B。

`listenfd` 继续作为监听 socket 使用；`accept` 返回新的连接 socket，用于该客户端连接上的 `read/write`。
:::

### 选择题 11（不定项）：协议 syntax 与 semantics

某协议规定请求格式为 `GET <key>\r\n`，并规定收到后返回 key 对应的 value 或 `NOT_FOUND`。以下哪些判断正确？

A. `GET <key>\r\n` 属于 syntax。  
B. “返回 key 对应的 value” 属于 semantics。  
C. syntax 只关心消息含义，不关心格式。

### 选择题 12（单选）：调度器如何重新获得控制权

单 CPU 上某用户线程一直计算，不做 I/O、不等待锁、不调用 `yield`。OS 通常依靠什么机制重新获得控制权？

A. 定时器中断。  
B. 用户线程主动调用普通函数。  
C. 编译器每行代码后自动调用调度器。  
D. `malloc` 一定触发调度。

:::tip 答案与解析
正确选项：A。

抢占式调度依赖外部异步事件，典型就是 timer interrupt。中断使控制权进入内核，内核可保存上下文并选择下一个线程。
:::

一次线程上下文切换通常需要保存或恢复哪些状态？

### 选择题 14（单选）：有界缓冲区的信号量顺序

以下哪个生产者代码顺序是标准正确模板？

A.
```c
P(mutex); P(empty); enqueue(); V(full); V(mutex);
```

B.
```c
P(empty); P(mutex); enqueue(); V(mutex); V(full);
```

C.
```c
P(full); P(mutex); enqueue(); V(mutex); V(empty);
```

D.
```c
P(mutex); enqueue(); P(empty); V(full); V(mutex);
```

:::tip 答案与解析
正确选项：B。

生产者先等待空槽 `P(empty)`，再进入队列临界区 `P(mutex)`；入队后先释放互斥锁，再通知满槽增加 `V(full)`。
:::

### 选择题 15（单选）：错误的 `P(mutex)` 顺序

若生产者在缓冲区已满时先执行 `P(mutex)`，再执行 `P(empty)`，最可能的问题是：

A. 没有问题，性能更好。  
B. 生产者阻塞时持有 `mutex`，消费者无法进入临界区取走元素，系统可能卡住。  
C. `empty` 会自动释放 `mutex`。  
D. 只会多消费一个元素。

:::tip 答案与解析
正确选项：B。

等待资源计数时不能持有保护队列的互斥锁，否则真正能改变条件的消费者无法进入临界区。
:::

课程中的 `test&set(address)` 原子原语语义是：

### 选择题 17（单选）：丢失唤醒

某锁实现中，线程发现锁忙后先加入等待队列，然后在真正 `sleep()` 之前重新打开中断。此时释放者可能唤醒它并把它放入 ready queue，但该线程随后继续执行 `sleep()`。这属于：

A. 队首阻塞。  
B. 丢失唤醒。  
C. DRF 违例。  
D. CFS 虚拟时间错误。

:::tip 答案与解析
正确选项：B。

唤醒已经发生，但线程随后又睡下，导致唤醒效果丢失。这正是 sleep/wakeup 原子性处理不当造成的 missed wakeup。
:::

### 选择题 18（不定项）：Mesa 管程中为什么用 `while`

在 Mesa 风格条件变量中，`cond_wait` 外通常应使用 `while` 重新检查条件。原因包括：

A. 被唤醒只表示条件可能成立过，不保证当前仍成立。  
B. 发 signal 的线程会继续执行，不会立即把 CPU 和锁交给等待者。  
C. 等待线程真正运行前，其他线程可能改变共享状态。  
D. 使用 `while` 是为了让代码慢一些，没有正确性意义。

:::tip 答案与解析

:::

### 选择题 19（单选）：读者/写者状态变量

写者优先读者/写者管程中，当前 `R1`、`R2` 正在读，随后 `W1` 到达等待，再随后 `R3` 到达。此时最合理的状态是：

A. `AR=3, WR=0, AW=0, WW=1`  
B. `AR=2, WR=1, AW=0, WW=1`  
C. `AR=2, WR=0, AW=1, WW=0`  
D. `AR=0, WR=3, AW=0, WW=1`

:::tip 答案与解析
正确选项：B。

已有两个活跃读者，所以 `AR=2`。W1 因读者活跃而等待，`WW=1`。写者优先策略下，R3 因已有等待写者而等待，`WR=1`。
:::

### 选择题 20（不定项）：单条件变量的风险

### 选择题 22（单选）：RR 数值计算

两个作业同时到达：`P1=5, P2=3`，RR 时间片 `q=2`，忽略上下文切换成本。若初始队列为 `P1, P2`，完成时刻为：

### 选择题 23（单选）：SRTF 推演

作业如下：

### 选择题 27（单选）：向量化死锁检测

系统有两类资源：

```text
Avail = [1, 0]
T1: Alloc = [1, 0], Request = [0, 1]
T2: Alloc = [0, 1], Request = [1, 0]
```

检测结果是：

A. 死锁，因为 T1 和 T2 互相等待。  
B. 无死锁，因为可先满足 T2，释放资源后再满足 T1。  
C. 死锁，因为 Avail 第二项为 0。  
D. 无法判断，因为检测算法不能处理多资源。

:::tip 答案与解析
正确选项：B。

`T2` 请求 `[1,0] <= [1,0]`，可模拟完成并释放 `[0,1]`，随后 `Avail=[1,1]`，T1 也可完成。因此无死锁。
:::

### 选择题 28（单选）：DRF 计算

总资源为 `<9 CPU, 18 GB>`。User 1 每个任务需求 `<1 CPU, 4 GB>`，User 2 每个任务需求 `<3 CPU, 1 GB>`。按 DRF，任务数应为：

A. `User1=3, User2=2`。  
B. `User1=2, User2=3`。  
C. `User1=4, User2=4`。  
D. `User1=1, User2=1`。

:::tip 答案与解析
正确选项：A。

User 1 的 dominant resource 是内存，dominant share 为 `4x/18`；User 2 的 dominant resource 是 CPU，dominant share 为 `3y/9`。令二者相等：

$$
4x/18 = 3y/9 \Rightarrow 2x = 3y \Rightarrow x = 1.5y
$$

约束为 `x+3y <= 9` 和 `4x+y <= 18`，取 `y=2`，`x=3`。
:::

### 选择题 29（单选）：Tiresias 2DAS

两个训练作业：

```text
A: 8 GPUs, 已运行 5 min
B: 2 GPUs, 已运行 30 min
```

按 Tiresias 的 2D attained service，谁的服务量更小？

A. A 更小，因为 A 运行时间短。  
B. B 更小，因为 B 的 2DAS 是 60 GPU-min，A 是 40 GPU-min。  
C. A 更小，因为 A 的 2DAS 是 40 GPU-min，B 是 60 GPU-min。  
D. 二者相等。

:::tip 答案与解析
正确选项：C。

2DAS 为 `#GPUs * executed time`。A 为 `8*5=40 GPU-min`，B 为 `2*30=60 GPU-min`。因此 A 的 attained service 更小。
:::

### 选择题 30（不定项）：ZygOS 与 FairRide 的客观语义

以下哪些说法正确？

A. ZygOS 试图结合 dataplane 低开销与 work-conserving 行为。  
B. ZygOS 的 remote syscall 用于让 remote core 在需要访问 home core 网络连接状态时回到 home core 处理。  
C. FairRide 的 blocking 只会影响作弊用户，绝不会影响诚实用户。  
D. FairRide 选择保留 isolation guarantee 和 strategy-proofness，并接受近似而非完美 Pareto efficiency。  
E. `p(n_j)=1/(n_j+1)` 中，`n_j` 越大，blocking 概率越大。

:::tip 答案与解析
正确选项：A、B、D。

ZygOS 的核心是低开销 dataplane 加 work stealing / work conservation。FairRide 的 blocking 可能影响诚实用户，因为系统不总能区分作弊和合法访问变化。`p(n_j)=1/(n_j+1)` 随 `n_j` 增大而减小，不是增大。
:::
