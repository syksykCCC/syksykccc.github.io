# Lec10 - 调度 1：概念与经典策略

## 学习目标
学完本讲后，你应当能够解释 CPU 调度到底在优化什么，比较 **FCFS**、**RR**、**严格优先级**、**SJF** 与 **SRTF**，能够用具体例子分析时间片取值的权衡，并理解为什么“平均完成时间最优”常常与“公平性”冲突。

## 1. 调度的核心问题
调度器反复回答同一个问题：**下一个应该把 CPU 分给哪个可运行线程？**

一个简化的内核视角是：
1. 若 ready queue 非空，选出一个线程控制块（TCB）。
2. 运行该线程。
3. 否则运行 idle 线程。

调度本质上是队列管理问题。

![调度的排队类比](./lec10_materials/scheduling_as_queue_analogy.png)

## 2. 假设与 CPU Burst 模型

### 2.1 简化假设
经典调度分析常以简化假设开局：
- 每个用户一个程序，
- 每个程序一个线程，
- 各程序彼此独立。

这些假设并不真实，但能让一阶策略分析变得可做。

### 2.2 CPU burst / I/O burst 交替
一个标准执行模型是程序在以下阶段之间交替：
- CPU burst，
- I/O burst，
- 再次 CPU burst。

经验上，CPU burst 的长度分布常常是**短 burst 占多数**。

![CPU burst 执行模型与分布](./lec10_materials/cpu_burst_execution_model_and_distribution.png)

每次调度决策通常都在决定：哪个作业拿到它“下一段 CPU burst”。

## 3. 调度目标与评价标准
课程强调三类目标：

1. **最小化完成时间（Completion Time）**
- 缩短作业完成所需的总耗时。
- 这是用户直接感知的指标（例如按键回显、编译时延、实时任务是否按时截止）。

2. **最大化吞吐量（Throughput）**
- 提升单位时间完成作业数。
- 吞吐量与完成时间相关，但并不等价。
- 吞吐量同时受低开销（如上下文切换）和资源利用率影响。

3. **公平性（Fairness）**
- 在用户/作业之间以某种公平方式分配 CPU。
- 更好的平均完成时间，往往可以通过“降低公平性”换来。

:::remark 问题：是否存在一个策略能同时把完成时间、吞吐量和公平性都做到最好？
通常不存在。三个目标之间常有拉扯，策略选择必须根据工作负载特征和服务目标来做取舍。
:::

## 4. 先来先服务（FCFS）

### 4.1 定义与直觉
**First-Come, First-Served (FCFS)** 也叫 FIFO，或“run until done”（现代语境下常理解为运行到线程阻塞）。

### 4.2 例子 1：到达顺序 `P1, P2, P3`
给定 burst 长度：
- `P1 = 24`
- `P2 = 3`
- `P3 = 3`

按 `P1 -> P2 -> P3` 调度：
- 等待时间：`P1=0, P2=24, P3=27`
- 平均等待时间：`(0+24+27)/3 = 17`
- 平均完成时间：`(24+27+30)/3 = 27`

![FCFS 示例：到达顺序 P1,P2,P3](./lec10_materials/fcfs_example_arrival_p1_p2_p3.png)

### 4.3 例子 2：到达顺序 `P2, P3, P1`
现在顺序变成“短-短-长”：
- 等待时间：`P1=6, P2=0, P3=3`
- 平均等待时间：`(6+0+3)/3 = 3`
- 平均完成时间：`(3+6+30)/3 = 13`

![FCFS 示例：到达顺序 P2,P3,P1](./lec10_materials/fcfs_example_arrival_p2_p3_p1.png)

这直接体现了 **head-of-line blocking（队首阻塞）**：短作业会被长作业压在后面。

### 4.4 FCFS 的优缺点
- 优点：
  - 非常简单，
  - 策略开销低。
- 缺点：
  - 对到达顺序非常敏感，
  - 对短作业有明显队首阻塞。

## 5. 时间片轮转（RR）

### 5.1 定义
**Round Robin (RR)** 通过时间片 `q` 引入抢占：
1. 每个就绪进程最多运行 `q` 时间单位，
2. 时间片到则被抢占并放到 ready queue 队尾。

若有 `n` 个可运行作业，每个作业约得到 `1/n` 的 CPU 份额，且每次连续运行不超过 `q`，并满足：
- **No process waits more than `(n - 1)q` time units.**

### 5.2 时间片的基本影响
- `q` 大 => 行为接近 FCFS。
- `q` 小 => 交错更强。
- `q` 过小 => 切换/切片开销高、推进碎片化。

:::remark 问题：时间片 `q` 应该怎么选？
`q` 的选择本质是在响应性与吞吐量之间做平衡。
- 太大（或趋近无穷）会退化到 FCFS，等待时延变差。
- 太小会把大量时间花在切换/切片上，吞吐下降。
- 工程上常见范围约为 `10ms` 到 `100ms`，再按负载调优。
:::

### 5.3 RR 计算示例（`q=20`）
给定：
- `P1=53`, `P2=8`, `P3=68`, `P4=24`

该调度得到：
- 等待时间：`P1=72, P2=20, P3=85, P4=88`
- 平均等待时间：`66.25`
- 平均完成时间：`104.5`

![RR 示例：q=20 甘特图](./lec10_materials/rr_example_quantum_20_gantt.png)

### 5.4 讨论题与解答
本讲提出两个关键问题：
- RR 在平均完成时间上是否总优于 FCFS？
- RR 的时间片更小是否总能改善平均完成时间？

:::remark 解答
两个问题答案都是否定的。
- 在某些工作负载上，RR 的平均完成时间会比 FCFS 更差。
- 更小的 `q` 在某些情况下确实有利于短作业，但也可能因为过度切片/切换导致完成时间变差。
:::

### 5.5 时间片反例
情形 A（更小 `q` 变好）：
- `T1=10`, `T2=1`
- `q=10` => 平均完成时间 `10.5`
- `q=5` => 平均完成时间 `8.5`

![减小时间片可能降低完成时间](./lec10_materials/quantum_reduction_can_decrease_completion_time.png)

情形 B（更小 `q` 变差）：
- `T1=1`, `T2=1`
- `q=1` => 平均完成时间 `1.5`
- `q=0.5` => 平均完成时间 `1.75`

![时间片过小可能升高完成时间](./lec10_materials/too_small_quantum_can_increase_completion_time.png)

### 5.6 内核实现要点
RR 实现通常保留 FIFO ready queue，并通过 timer interrupt 实现时间片到期抢占，同时必须对队列与状态迁移做严格同步。

![RR 内核中的 timer interrupt 流程](./lec10_materials/rr_kernel_timer_interrupt_flow.png)

## 6. FCFS 与 RR 的更强对比
一个典型反例假设显式上下文切换成本为 0：
- 10 个作业，
- 每个作业都需 100 秒 CPU，
- RR 时间片为 1 秒，
- 所有作业同时到达。

完成时间模式变为：
- FCFS：`100, 200, ..., 1000`
- RR：`991, 992, ..., 1000`

因此 RR 的平均完成时间明显更差，即使总体结束时刻相同。

![FCFS 与 RR 完成时间反例](./lec10_materials/fcfs_rr_completion_time_counterexample.png)

更大的量化表（不同 `q` 与不同 FCFS 顺序）进一步说明：
- 某些负载下，最佳次序的 FCFS 可能显著更优，
- RR 的表现对 `q` 和负载构成高度敏感。

![时间片敏感性：等待时间与完成时间总表](./lec10_materials/quantum_sensitivity_wait_completion_table.png)

## 7. 严格优先级调度

### 7.1 策略
**Strict Priority Scheduling** 总是先运行最高优先级队列（每个队列内部仍可用 RR）。

![严格优先级调度结构](./lec10_materials/strict_priority_scheduling_structure.png)

### 7.2 问题
- 饥饿（Starvation）：
  - 低优先级作业可能长期得不到 CPU。
- 优先级反转（Priority Inversion）：
  - 高优先级任务等待低优先级任务持有的锁，
  - 中优先级任务插入运行会进一步放大延迟。

### 7.3 常见缓解方向
通过动态优先级调整（aging、交互性/锁行为/burst 行为等启发式）来降低极端不公平。

## 8. 公平性讨论
跨队列严格固定优先级会让长作业不公平。常见公平化方向包括：

1. 给每个队列固定 CPU 份额。
2. 提升长期得不到服务作业的优先级。

两者都有副作用：
- 固定份额可能与队列规模不匹配，
- 激进的优先级提升容易变成经验性参数、在过载时不稳定。

核心权衡仍是：
- 公平性提升通常要牺牲平均完成时间最优性。

## 9. 如果能预知未来：SJF 与 SRTF

### 9.1 定义
**Shortest Job First (SJF)**：
- 选择总计算量最小的作业先跑。
- 也常称 STCF（shortest time to completion first）。

**Shortest Remaining Time First (SRTF)**：
- SJF 的抢占版本：新到作业若剩余时间更短，立即抢占当前作业。
- 也常称 SRTCF。

### 9.2 最优性结论
- SJF 在非抢占策略中对平均完成时间最优。
- SRTF 在抢占策略中对平均完成时间最优。
- 由于 SRTF 至少不差于 SJF，SRTF 是更强的参考基线。

### 9.3 直观比较
- 若所有作业等长，SRTF 会退化为接近 FCFS 的行为。
- 若作业长度差异明显，SRTF 能避免短作业被长作业长期压住。

:::remark 问题：我们能否总是复现“最优次序”的 FCFS？
只有在能精确预知未来运行时长时才行。真实系统里的运行时预测并不准确，因此这种“先知式排序”通常无法直接实现。
:::

## 10. 示例：为什么 SRTF 有利于交互/I/O 型任务
负载设定：
- `A`、`B`：长 CPU 密集任务，
- `C`：I/O 密集循环（`1ms CPU + 9ms disk I/O`）。

定性结果：
- FCFS 可能让 `A/B` 长时间占住 CPU，`C` 的磁盘利用率很低（图中极低）。
- RR 粗粒度时间片可改善利用率，但会带来较多唤醒/切换交互。
- SRTF 能更好照顾短 burst，通常可维持高利用率且推进更平滑。

![SRTF 时间线与磁盘利用率对比](./lec10_materials/srtf_workload_timeline_disk_utilization.png)

## 11. SRTF 的局限与工程现实
SRTF 的两个主要现实问题：
1. 若短作业持续到达，长作业可能饥饿。
2. 需要预测未来运行时间，而预测本身困难且有噪声。

有些系统让用户申报运行时长，但这既可能被利用，也常常对诚实用户都很难准确。

因此在工程实践里：
- SRTF 更像重要的性能标尺（yardstick），
- 真实系统常做近似，并叠加公平性机制。

## 12. 本讲核心结论
- **FCFS** 简单，但有队首阻塞。
- **RR** 提升响应与公平，但高度依赖时间片调参。
- **严格优先级** 能表达重要性，但有饥饿与优先级反转风险。
- **SJF/SRTF** 理论上最优平均完成时间，但依赖难以获得的未来信息，且可能不公平。
- 真实调度器本质上是围绕负载与服务目标的折中设计。

## 附录 A：Exam Review

### A.1 必背定义
- **FCFS**：按到达顺序执行（通常直到阻塞）。
- **RR**：时间片抢占轮转。
- **严格优先级**：总先运行最高优先级可运行任务。
- **SJF**：选总工作量最短作业。
- **SRTF**：选剩余工作量最短作业（可抢占）。

### A.2 快速对比清单
- 完成时间最优性：
  - 理论最优：SRTF（非抢占场景下 SJF 最优）。
- 公平性：
  - 基线公平性通常 RR 更强（前提是时间片合适）。
- 饥饿风险：
  - 严格优先级与 SRTF（无保护时）较高。
- 调参敏感性：
  - RR 对时间片非常敏感。

### A.3 常考简答题
1. 为什么 RR 不一定比 FCFS 平均完成时间更好？
2. 为什么时间片过小会拖慢性能？
3. FCFS 的队首阻塞是什么？
4. 严格优先级为什么会导致饥饿？
5. 为什么 SRTF 很难在真实系统中精确落地？

### A.4 计算模板
- 单个进程等待时间：
  - 该进程每次获得 CPU 之前等待区间的总和。
- 单个进程完成时间：
  - 完成时刻减到达时刻。
- 平均指标：
  - 对所有作业取算术平均。

### A.5 常见错误
- 误以为 RR 在平均完成时间上总优于 FCFS。
- 把“吞吐量最大化”和“完成时间最小化”简单等同。
- 设计严格优先级或最短优先时忽略饥饿。
- 讨论 SRTF 时忽略预测误差与可实现性。
